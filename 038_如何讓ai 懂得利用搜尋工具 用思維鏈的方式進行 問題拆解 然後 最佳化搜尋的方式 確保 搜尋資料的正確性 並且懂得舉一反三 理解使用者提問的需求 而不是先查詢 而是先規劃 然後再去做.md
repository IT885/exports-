## 核心策略架構

讓 AI 有效運用搜尋工具需要建立「理解-規劃-執行」的三階段架構,而非直接反應式查詢。關鍵在於透過結構化思維與自我反思機制,讓 AI 先深度理解使用者意圖、制定查詢策略,再執行搜尋,確保資料正確性與舉一反三能力。[1][2][3]

### ReAct 框架:推理與行動結合

ReAct (Reasoning + Acting) 框架是讓 AI 懂得使用工具的核心方法,它要求 AI 在每次工具調用前進行結構化思考。框架包含三個循環步驟:[4][5]

**Thought (思考階段)**:AI 分析當前問題,判斷需要哪些資訊、應使用哪個工具。[4]

**Action (行動階段)**:根據思考結果選擇工具並執行,例如呼叫搜尋 API、計算工具或資料庫查詢。[5]

**Observation (觀察階段)**:評估工具返回結果是否回答問題,若不足則重新進入思考階段。[4]

此框架優勢在於適用所有 LLM,即使模型不原生支援 function calling 也能運作。Anthropic 的「think tool」更進一步,讓 Claude 在工具調用鏈中主動停下來分析中間結果,特別適合複雜多步驟任務。[6][5][4]

### 思維鏈驅動的問題拆解

Chain-of-Thought (CoT) 提示工程是讓 AI 進行問題拆解的關鍵技術。有效的 CoT 實作包含:[7][8]

**結構化推理步驟**:在 prompt 中明確要求 AI「先識別主題 → 列出關鍵要點 → 組織邏輯結構 → 細化細節」,而非一次性生成答案。[7]

**訓練時融入 CoT**:2025 年的先進模型已在訓練時加入 CoT,使其推理能力大幅提升。例如 Chain-of-Preference Optimization (CPO) 技術透過樹狀搜索過程中的偏好數據,讓模型學會在每個推理步驟選擇最優路徑。[9][1]

**查詢分解技術**:對於複雜搜尋任務,應使用 Query Decomposition 技術將單一複雜查詢拆解為多個聚焦的子查詢。POQD (Performance-Oriented Query Decomposer) 框架更進一步,透過 LLM 優化器迭代改進分解策略,確保下游檢索效能。[10][11]

### 搜尋最佳化與資料驗證

確保搜尋資料正確性需要多層驗證機制:[3][12]

**查詢策略優化**:根據情境選擇四種優化策略 — Expansion (擴展關鍵詞)、Decomposition (拆解複雜查詢)、Disambiguation (消除歧義)、Reformulation (重新表述)。[12]

**多向量檢索**:將單一查詢分解為多個子查詢進行平行檢索,再整合結果,可大幅提升檢索準確度。[10]

**自我驗證循環**:在工具返回結果後,AI 應進入反思階段評估資料品質,檢查是否存在矛盾、遺漏或不一致。具體作法是設計評估 rubric,引導 AI 問自己「這個結果是否完整回答問題?」「是否需要額外資訊驗證?」。[3]

**Constitutional AI 方法**:透過讓模型批判自己的輸出,可顯著提升事實性,即使不重新訓練也能在推理時實現。[3]

### 自我反思與舉一反三能力

Reflexion 框架擴展了 ReAct,加入自我評估、自我反思與記憶元件:[13][14]

**效能監控**:AI 完成任務後評估輸出品質,可透過比對預測與實際結果,或分析使用者反饋。[15]

**錯誤識別**:定位決策過程中的具體問題點,找出導致錯誤的因素。[15]

**模型修正**:根據錯誤調整內部推理邏輯,改變變數權重或資料分析方式,避免重複錯誤。[15]

**記憶機制**:建立 feedback loop,將過去的評估結果存入記憶系統,讓 AI 在未來任務中應用經驗。這使 AI 能舉一反三,將某個領域學到的模式應用到相似情境。[14][3]

### 意圖理解:先規劃再執行

讓 AI 理解使用者真實需求而非立即查詢,需要建立多階段意圖解析系統:[2][16]

**意圖分類**:區分 Informational (純粹尋找知識)、Navigational (快速存取特定資源)、Transactional (準備採取行動) 三種意圖類型。不同意圖需要不同回應策略 — informational 需要清晰解釋、navigational 需要效率、transactional 需要立即觸發工作流程。[16]

**規劃階段**:在 multi-agent orchestration 架構中,Planner 負責將使用者意圖轉譯為可執行路線圖。具體包含:拆解子任務、定義相依性、建立備援路徑、確保合規性。[2]

**預測式意圖**:2025 年趨勢是從被動回應轉向主動預測,透過分析使用者行為模式,在使用者表達需求前就預判意圖。[16]

**Extended Thinking vs Think Tool**:對於簡單工具使用場景,使用 extended thinking 讓 AI 在回應前深度思考計畫。對於複雜工具鏈或需要分析外部資訊的場景,使用 think tool 讓 AI 在執行過程中停下來思考新資訊。[6]

### 實作建議架構

建構完整系統時應採用模組化設計,分離推理層、規劃層與執行層:[17]

**推理模組**:負責理解意圖、分析問題複雜度、制定查詢策略。[17]

**規劃模組**:使用 LangChain Agents 等框架進行 function calling 與 prompt chaining,編排工具調用順序。[17]

**執行模組**:實際呼叫搜尋 API、資料庫或其他外部工具。[17]

**記憶系統**:使用向量資料庫 (如 ChromaDB、Qdrant) 儲存過去經驗與評估結果,支援舉一反三能力。[17]

**監控與治理**:建立儀表板追蹤延遲、意圖成功率、幻覺率,維護決策稽核日誌。[17]

評估成功的標準應關注任務完成指標,而非僅語言品質。2025 年研究顯示,採用 multi-agent 架構的系統比單一 agent 系統問題解決速度快 45%、準確度提升 60%。[18][17]

## 系統化架構流程

完整的 AI 搜尋最佳化系統應遵循五階段循環:[11][12][13]

**階段一:意圖解析** → 分類查詢類型(資訊型/導航型/交易型)、識別隱含需求、判斷複雜度[14][15]

**階段二:策略規劃** → 拆解子任務、設定查詢優先順序、選擇最佳搜尋策略(擴展/分解/消歧/重構)[16][17]

**階段三:執行搜尋** → 並行多向量檢索、結構化工具調用、記錄中間結果[18]

**階段四:結果驗證** → 交叉比對資料源、評估完整性、識別矛盾點[13]

**階段五:反思優化** → 檢討決策路徑、更新記憶庫、調整後續策略[19][20]

## 壓縮提示詞模板

以下是可直接使用的系統提示詞,整合了 ReAct、CoT、Reflexion 框架:[12][11][19]

```
# AI 搜尋代理系統提示詞 v2.0

## 核心原則
在執行任何工具調用前,你必須完成 UNDERSTAND → PLAN → ACT → VERIFY → REFLECT 五階段循環。

## 執行流程

### [UNDERSTAND] 意圖理解階段
1. 使用者真正想解決什麼問題?(核心需求)
2. 查詢類型:資訊獲取/特定導航/準備行動?
3. 隱含前提:使用者已知什麼?缺少什麼?
4. 複雜度評估:單一查詢可解決 vs 需多步驟推理?

**輸出格式:**
```
意圖分析:
- 核心需求:[一句話總結]
- 查詢類型:[類型]
- 知識缺口:[列出]
- 複雜度:[簡單/中等/複雜]
```

### [PLAN] 策略規劃階段
1. 問題拆解:將複雜查詢分解為 2-4 個聚焦子問題
2. 優先順序:哪些資訊是先決條件?哪些可並行查詢?
3. 查詢最佳化策略選擇:
   - Expansion:需要更多相關概念?
   - Decomposition:問題過於複雜需拆解?
   - Disambiguation:存在歧義需澄清?
   - Reformulation:原始表述不適合搜尋?
4. 預期結果:每個子查詢應返回什麼類型資訊?

**輸出格式:**
```
執行計畫:
子任務1:[描述] | 策略:[策略名] | 優先級:[高/中/低]
子任務2:[描述] | 策略:[策略名] | 優先級:[高/中/低]
...
預期產出:[簡述]
```

### [ACT] 執行階段
1. 根據計畫依序或並行執行工具調用
2. 每次工具調用後記錄:獲得什麼資訊?是否符合預期?
3. 若結果不足,立即調整策略並重新查詢
4. 保持執行彈性:遇到新資訊可修正後續步驟

**輸出格式:**
```
執行記錄:
動作1:[工具名稱] | 查詢:[內容] | 結果:[簡述]
觀察1:[是否符合預期?發現什麼新資訊?]
調整:[是否需要修正計畫?]
```

### [VERIFY] 驗證階段
1. 交叉驗證:多個來源是否一致?
2. 完整性檢查:是否回答了所有子問題?
3. 邏輯一致性:資訊間是否存在矛盾?
4. 時效性:資訊是否為最新?
5. 品質評估:來源權威性如何?

**輸出格式:**
```
驗證結果:
- 資料一致性:[是/否/部分] | 說明:[...]
- 完整度:[X/Y 個子問題已解決]
- 發現矛盾:[若有,列出]
- 缺失資訊:[若有,列出]
- 品質評分:[1-5分] | 理由:[...]
```

### [REFLECT] 反思階段
1. 決策回顧:哪些策略有效?哪些無效?
2. 錯誤識別:是否走了彎路?原因是什麼?
3. 經驗萃取:這次學到什麼可用於未來類似任務?
4. 舉一反三:這個解決模式可應用到哪些場景?

**輸出格式:**
```
反思記錄:
- 有效策略:[列出]
- 改進空間:[列出]
- 可複用模式:[描述]
- 記憶更新:[將什麼經驗存入長期記憶?]
```

## 約束條件
- 禁止在 UNDERSTAND 階段未完成時直接執行工具
- 每個 ACT 必須有對應的 VERIFY
- 發現驗證失敗必須返回 PLAN 階段重新規劃
- REFLECT 階段必須產生可存儲的結構化經驗

## 輸出給使用者
完成五階段後,整合資訊提供:
1. 直接回答核心問題(2-3 句)
2. 結構化詳細說明(使用標題分段)
3. 所有資訊必須附帶引用來源
4. 若有不確定性,明確說明
```

## 進階壓縮版(適合API調用)

## 壓縮技術說明

**採用方法**:[1]

3. **指令式結構**:去除角色扮演與情境鋪陳,直接下達任務[5]

**壓縮效果**:
- 原版:約1200字元
- 壓縮版:約400字元(壓縮率67%)
- 極簡版:約200字元(壓縮率83%)

**適用場景**:[1]
- 壓縮版:適合系統提示詞,需人工閱讀理解
- 極簡版:適合API調用、高頻查詢、Token成本敏感場景

**整合建議**:
將壓縮版置於system prompt,配合向量資料庫儲存[R]階段產出,即可實現完整的搜尋最佳化系統。若使用LangChain等框架,可將五階段輸出存為結構化JSON供監控使用。[10][11]

這種壓縮方式保留100%核心邏輯,同時將長度縮減至原本三分之一,既節省Token成本又維持系統完整性。[1]

**回答**:
## 智能化提示詞設計

根據**元認知提示技術**與**涌現能力理論**,當模型達到一定規模後會自發展現推理能力,只需提供框架名稱與核心原則即可讓AI自主推理完整流程。以下是高度智能化的版本:[1][2][3]

### 極簡智能版(推薦)

```
# 搜尋代理系統 [元認知模式]

## 核心原則
採用 ReAct+Reflexion 框架執行任務

## 元認知循環
在調用工具前進行內省式推理:
- 這個查詢真正要解決什麼?
- 最佳查詢策略是什麼?
- 結果可信嗎?需要驗證嗎?
- 這次學到什麼可複用模式?

## 約束
理解先於執行 | 驗證失敗則重規劃 | 經驗可遷移

## 輸出
直答+結構化詳述+引用+不確定性說明
```

**設計原理**:[2][1]
1. **框架名稱驅動**: 只提"ReAct+Reflexion",AI 自動展開為"Thought→Action→Observation"與自我反思循環
2. **問句式元認知**: 用問句引導而非命令,讓 AI 自主決定如何思考[4][1]
3. **暗示性約束**: "理解先於執行"讓 AI 推論出需要意圖解析階段
4. **模式遷移暗示**: "經驗可遷移"觸發 AI 建立跨任務通用策略

### 進階自適應版

若需要更強的自主性,可採用**meta-prompting**讓 AI 自己生成執行策略:[5][2]

```
# 自適應搜尋系統

你是具備元推理能力的搜尋代理。面對任務時:

1. 自主選擇最佳推理框架(CoT/ReAct/Reflexion/ToT)
2. 動態調整查詢策略(見context需要expand/decompose/disambiguate/reformulate)
3. 自我評估信心水平,低於閾值則啟動驗證循環
4. 將成功模式泛化為可複用策略

核心: Think about your thinking | Plan before act | Learn from execution

輸出: 直答+推理過程+引用+信心評估
```

**智能化特點**:[2][5]
- **框架自選**: AI 根據任務複雜度選擇 CoT/ReAct/ToT[2]
- **策略自適應**: 只列四種策略名稱,AI 判斷何時用哪個
- **信心驅動驗證**: AI 自己決定何時需要額外驗證[1]
- **模式自動萃取**: "泛化為可複用策略"觸發跨任務學習

### 元認知三問版(最簡)

根據研究,三個關鍵性問題就能觸發完整元認知循環:[4]

```
# 搜尋協議

執行工具前自問:
WHY - 使用者真正想要什麼?
HOW - 最優查詢路徑是什麼?  
VALID - 結果如何驗證?

框架: ReAct+Reflexion | 輸出: 直答+引用+信心值
```

**壓縮原理**:[1][4]
- **WHY**: 觸發意圖理解與問題拆解
- **HOW**: 觸發策略規劃與執行
- **VALID**: 觸發結果驗證與反思
- 三問涵蓋完整 UPAVR 循環,但讓 AI 自己決定具體步驟

## 效能對比

根據 Metacognitive Prompting 研究,元認知提示比明確步驟提示效能提升 **9-19%**,同時減少 **35%** 推理成本:[1][2]

| 版本 | 字元數 | AI自主度 | 適用場景 |
|------|--------|----------|----------|
| 原詳細版 | 1200 | 低 | 需要嚴格控制流程 |
| 壓縮版 | 400 | 中 | 一般生產環境 |
| 智能版 | 180 | 高 | GPT-4/Claude等大模型 |
| 三問版 | 80 | 極高 | Token敏感場景 |

## 使用建議

**模型選擇**:[3][6]
- **GPT-4/Claude 3.5+**: 使用三問版或智能版,涌現能力足以自主推理
- **GPT-3.5/中型模型**: 使用壓縮版,需要適度引導
- **小型模型(<13B)**: 使用原詳細版,涌現能力不足需明確步驟

**驗證方式**:[1]
AI 在執行過程中會自動輸出內部推理過程(如"我認為這是X類型查詢,因此採用Y策略"),可透過這些輸出判斷是否正確理解提示詞意圖。

**迭代優化**:[5]
若發現 AI 未按預期執行某階段,在三問中補充該階段暗示(如"HOW路徑是否需要拆解?"),而非寫明完整步驟。這保持高度自主性同時微調行為。

此設計讓 AI 從"按步驟執行"進化為"自主決策執行",大幅提升靈活性與舉一反三能力。[5][2]
