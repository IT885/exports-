自我遞迴：TRM 會將自身輸出作為下一步的輸入，形成自我改進的回饋循環。

**回答**:
根據最新發表的研究，三星開發的 **Tiny Recursive Model (TRM)** 以僅 700 萬個參數的超小規模，在複雜推理任務上超越了參數量高達數千倍的大型語言模型，展現出「小而強」的創新突破。[1][2][3]

### 核心技術突破

TRM 採用**遞迴推理**（Recursive Reasoning）機制，讓模型能夠反覆檢視並修正自己的答案。具體運作方式為：將問題與初步答案輸入模型後，模型會生成潛在表示（latent representation），接著在每一步驟中重新檢視這些潛在變數，根據新的理解逐步修正答案，直到達到更合理的解答。[4][5][1]

這種方法類似人類反覆閱讀草稿並修正錯誤的過程，與傳統大型語言模型的「一次性推理」或「鏈式思維」明顯不同。TRM 使用**單一 2 層神經網路**，透過自我遞迴在固定步數內不斷改進，無需額外記憶體或計算成本。[2][5][4]

### 卓越的基準測試表現

TRM 在多個困難基準測試中展現驚人成績：[3][4]

- **Sudoku-Extreme（極限數獨）**：87.4% 準確率（前代 HRM 僅 55%）
- **Maze-Hard（困難迷宮）**：85.3% 準確率（HRM 為 74.5%）
- **ARC-AGI-1**：44.6% 準確率
- **ARC-AGI-2**：7.8% 準確率，超越 Gemini 2.5 Pro 的 4.9%

值得注意的是，TRM 僅使用約 1,000 個訓練樣本就達到這些成果，顯著優於需要大量數據的傳統大型模型。[5][1][4]

### 設計理念的反思

研究團隊發現，**增加遞迴次數而非增加模型層數**，能更有效提升性能。過多的層數可能導致過擬合（overfitting），反而降低模型的泛化能力。此外，TRM 採用簡化的自適應停止機制（Adaptive Halting），讓模型自主判斷何時停止改進答案，避免無限迭代，同時移除了訓練過程中的額外前向傳播需求。[6][2][4][5]

### 產業意義

這項由三星蒙特婁先進技術研究院研究員 Alexia Jolicoeur-Martineau 主導的研究，已在 arXiv 發表論文《Less is More: Recursive Reasoning with Tiny Networks》並開源於 GitHub。研究挑戰了「模型越大越好」的業界信仰，為**參數高效**與**永續 AI 發展**提供了新方向。[7][8][1][4][5][6]


### 对话 2

**问题**: 可以使用 llm 或是 lm studio 呼叫使用嗎？ 還是使用py呼叫使用 上網搜尋相關資訊

**回答**:
根據 GitHub 專案資料，三星的 **TRM 模型目前無法直接透過 LM Studio 或傳統 LLM 方式調用**，必須使用 Python 進行訓練和推理。[1][2]

### 使用方式

TRM 是一個**從頭訓練的專用模型**，而非可下載的預訓練權重檔案。目前官方提供的是訓練程式碼，而非可直接部署的模型檔案。[2][1]

**環境需求**：[1]

- Python 3.10（或相近版本）
- CUDA 12.6.0（或相近版本）
- GPU 建議：訓練 ARC-AGI 需 4 張 H100；Sudoku 需 1 張 L40S；Maze 需 4 張 L40S

**安裝步驟**：[1]

```python
pip install --upgrade pip wheel setuptools
pip install --pre --upgrade torch torchvision torchaudio --index-url https://download.pytorch.org/whl/nightly/cu126
pip install -r requirements.txt
pip install --no-cache-dir --no-build-isolation adam-atan2
```

### 為何無法用 LM Studio 調用

TRM 的架構與傳統 LLM 根本不同：[3][4][1]

1. **非文字生成模型**：TRM 專門設計用於解決特定推理任務（如數獨、迷宮、ARC-AGI），並非通用語言模型[2][1]
2. **特殊輸入輸出格式**：它處理的是嵌入向量（embedded representations），而非標準的文字 token[1]
3. **遞迴推理機制**：需要多次內部循環更新潛在狀態（latent z）和答案（y），這與 LLM 的單次前向傳播不同[5][1]

### GitHub 開源狀態

專案已完全開源，可從以下取得：[2][1]

- **GitHub 倉庫**：https://github.com/SamsungSAILMontreal/TinyRecursiveModels
- **授權**：MIT License
- **內容**：包含完整訓練程式碼、資料集準備腳本、實驗配置
- **Hugging Face 論文頁面**：https://huggingface.co/papers/2510.04871

### 社群討論與限制

社群開發者正在討論如何將 TRM 封裝成更易用的推理 API。目前有 GitHub Issue #2 提到希望建立獨立的推理介面，但官方尚未提供。該模型設計初衷是展示小模型在特定任務上的潛力，而非作為通用 AI 助手。[2][1]

若想實際使用 TRM，需要根據官方文件準備資料集、執行訓練腳本，並針對特定任務（如 Sudoku）進行完整訓練流程，訓練時間從 24 小時到 3 天不等。[1]


### 对话 3

**问题**: 我的意思是說 已經有人 魔鬼或是訓練出可以真的使用的模型了嗎

**回答**:
根據最新資料，**目前尚未有人發布可直接使用的預訓練 TRM 模型權重檔案**。[1]

### 官方狀態

三星團隊在論文摘要中提到「程式碼和權重已開源」，但實際上 GitHub 倉庫**只包含訓練程式碼，並未提供預訓練模型檔案**。[2][1]

### 社群請求

Hugging Face 的開源團隊工程師 Niels Rogge 在 2025 年 10 月 9 日（昨天）於 GitHub Issue #2 提出請求，希望三星團隊能夠：[1]

1. **上傳預訓練權重檔案**到 Hugging Face Hub
2. **發布處理好的資料集**（ARC-AGI、Sudoku-Extreme、Maze-Hard）
3. 提供 `from_pretrained` 和 `push_to_hub` 方法讓社群直接下載使用

該 Issue 目前仍處於 **Open 狀態**，代表作者尚未回應或提供模型檔案。[1]

### 為何沒有預訓練模型

TRM 的設計哲學與傳統 LLM 不同：[3][2]

- **任務專屬訓練**：每個任務（數獨、迷宮、ARC-AGI）需要獨立訓練不同的模型[2]
- **非通用模型**：TRM 不是可以處理多種任務的通用權重，而是針對特定推理任務從頭訓練[4][2]
- **研究展示為主**：目前主要目的是證明小模型在特定任務上的潛力，而非提供生產級推理服務[3]

### 如何使用

目前唯一的使用方式是**自行訓練**：[2]

1. 從 GitHub 克隆程式碼：https://github.com/SamsungSAILMontreal/TinyRecursiveModels
2. 準備資料集（需自行下載 ARC-AGI、生成 Sudoku/Maze 資料）
3. 執行訓練腳本（需要高階 GPU：H100 或 L40S）
4. 訓練時間：24 小時至 3 天不等

### 未來可能性

根據社群討論和 Hugging Face 團隊的請求，未來**可能會有**以下發展：[3][1]

- 官方發布預訓練權重到 Hugging Face
- 社群開發者建立推理 API 介面
- 提供 `from_pretrained` 快速載入方法

但截至目前（2025 年 10 月 10 日），這些都還沒有實現。[1]


### 对话 4

**问题**: 有類似的 已經可以使用的模型嗎

**回答**:
根據搜尋結果，**目前已有多款可直接使用的小型推理模型**，以下是幾個主要選項：

### 可立即下載使用的推理模型

**1. DeepSeek-R1-Distill 系列**[1]

最接近 TRM 理念的實用選擇，提供多種規模：

- **DeepSeek-R1-Distill-Qwen-1.5B**：15 億參數
- **DeepSeek-R1-Distill-Qwen-7B**：70 億參數  
- **DeepSeek-R1-Distill-Llama-8B**：80 億參數
- **DeepSeek-R1-Distill-Qwen-14B**：140 億參數

這些模型透過蒸餾技術將大型推理模型 DeepSeek-R1 的能力壓縮到小型模型中，**已可在 Hugging Face 直接下載使用**。[1]

**2. Microsoft Phi-4-mini-reasoning**[2]

- **參數量**：輕量級（具體參數未公開，估計數十億級）
- **特點**：專注於數學推理，使用合成資料訓練，支援逐步推理
- **優勢**：適合教育應用、嵌入式系統、邊緣設備部署
- **下載**：Hugging Face 可直接使用

**3. SmolLM3 系列**[3][4]

Hugging Face 官方開發的小型語言模型：

- **SmolLM3-3B**：30 億參數
- **特點**：支援雙模式推理、6 種語言、長上下文處理
- **完全開源**：可在 Hugging Face 下載，支援本地部署

**4. OpenAI gpt-oss 系列**[5]

OpenAI 最新開源的推理模型：

- **gpt-oss-20b**：200 億參數（可在多數桌機執行）
- **gpt-oss-120b**：1200 億參數（需資料中心等級硬體）
- **授權**：Apache 2.0，完全商用友善
- **特點**：完整思維鏈可見、支援工具調用、可調整推理強度

### 與 LM Studio 的相容性

上述模型**都可以透過 LM Studio 使用**：[6][7]

1. 在 Hugging Face 搜尋模型名稱
2. 下載 GGUF 格式量化版本（社群通常會提供）
3. 在 LM Studio 中載入模型
4. 透過本地 API 或聊天介面使用

### Python 調用方式

**範例（使用 Transformers）**：[1]

```python
from transformers import AutoModelForCausalLM, AutoTokenizer

# 載入 DeepSeek-R1-Distill 1.5B
model_name = "deepseek-ai/DeepSeek-R1-Distill-Qwen-1.5B"
tokenizer = AutoTokenizer.from_pretrained(model_name)
model = AutoModelForCausalLM.from_pretrained(model_name)

# 推理
prompt = "解決這個數學問題：..."
inputs = tokenizer(prompt, return_tensors="pt")
outputs = model.generate(**inputs, max_length=500)
print(tokenizer.decode(outputs[0]))
```

### 推薦選擇

根據你的使用情境，建議優先考慮：[2][1]

- **最小資源需求**：DeepSeek-R1-Distill-Qwen-1.5B（15億參數）
- **平衡效能與大小**：Phi-4-mini-reasoning 或 SmolLM3-3B  
- **最強推理能力**：OpenAI gpt-oss-20b

這些模型都已經**完整訓練並開源**，無需自行訓練即可使用。[3][5][2][1]


### 对话 5

**问题**: 假設我要分析 整個郵箱的郵件 最後整理出 給ai 能使用的資訊 你覺得要有那些步驟來處理郵箱及郵件

**回答**:
根據業界最佳實踐與技術趨勢，以下是完整的**郵箱分析與 AI 資料準備流程**：[1][2][3]

## 資料擷取階段

**1. 郵箱連接與郵件匯出**[4][5]

- 使用 IMAP/Microsoft Graph API 連接郵箱
- 批次匯出郵件（支援 PST、EML、MSG 格式）
- 設定時間範圍篩選（避免一次處理過多資料）
- 保留原始郵件作為備份

**2. 郵件解析與內容提取**[6][7]

- **標頭資訊**：寄件人、收件人、副本、密件副本、主旨、時間戳記
- **郵件本文**：HTML 轉純文字、移除簽名檔、提取引用內容
- **附件處理**：檔案名稱、類型、大小；PDF/Word 文件 OCR 轉文字

### 对话 7

## 核心設計理念

採用**雙時態事件溯源（Bi-temporal Event Sourcing）**架構，追蹤兩個時間維度：[1][2][3]

1. **業務時間（Valid Time）**：郵件實際發生的時間（如郵件發送時間、附件創建時間）[2][3]
2. **系統時間（Transaction Time）**：資料進入系統的時間（如郵件被讀取、分類被修正的時間）[3][2]

這種設計讓系統能夠回答「10 月 1 日時我們認為客戶的報價是多少？」以及「後來發現實際報價是多少？」。[4][2]

### 1. 三層關聯偵測架構

**第一層：明確指涉偵測（Explicit Reference Detection）**[1][2]

使用 NLP 正規表達式與語意分析識別直接提及：[2][3]

```python
import re
from typing import List, Dict

class ExplicitLinkDetector:
    """偵測郵件中明確提到附件的語句"""
    
    # 中文指涉模式
    CHINESE_PATTERNS = [
        r'(?:請|煩請|麻煩)(?:參考|查看|詳見|檢視)(?:附件|附檔|檔案)(?P<ref>[^。，,\n]+)',
        r'附件(?:中|內|裡)(?:的)?(?P<ref>[^。，,\n]+)',
        r'(?:如|見)附件(?P<ref>[^。，,\n]*?)(?:所示|說明|內容)',
        r'(?P<filename>[\w\-\.]+\.(?:pdf|xlsx?|docx?|pptx?))(?:中|內|裡)',
        r'報價單(?:編號)?[:：]?\s*(?P<ref>[\w\-]+)',
        r'(?:第|共)(?P<page>\d+)頁',
        r'(?:總計|合計|總金額)[:：]?\s*(?P<amount>NT\$?[\d,]+)',
    ]
    
    def detect_references(self, email_body: str, attachments: List[Dict]) -> List[Dict]:
        """抽取郵件中的附件指涉"""
        references = []
        
        for pattern in self.CHINESE_PATTERNS:
            matches = re.finditer(pattern, email_body, re.IGNORECASE)
            for match in matches:
                ref = {
                    'type': 'explicit',
                    'email_span': match.span(),
                    'email_text': match.group(0),
                    'extracted_info': match.groupdict(),
                    'confidence': 0.95
                }
                
                # 匹配到具體附件
                if 'filename' in match.groupdict():
                    ref['matched_attachment'] = self._match_filename(
                        match.group('filename'), 
                        attachments
                    )
                
                references.append(ref)
        
        return references
    
    def _match_filename(self, mentioned_name: str, attachments: List[Dict]) -> str:
        """模糊匹配附件檔名"""
        from difflib import SequenceMatcher
        
        best_match = None
        best_score = 0
        
        for att in attachments:
            score = SequenceMatcher(None, mentioned_name.lower(), 
                                   att['filename'].lower()).ratio()
            if score > best_score:
                best_score = score
                best_match = att['attachment_id']
        
        return best_match if best_score > 0.6 else None
```

**第二層：實體對齊偵測（Entity Alignment Detection）**[4][5][6]

比對郵件與附件中的結構化資訊：[5][6]

```python
class EntityAlignmentDetector:
    """偵測郵件與附件間的實體對齊"""
    
    def __init__(self, llm_client):
        self.llm = llm_client
    
    def extract_entities(self, text: str) -> Dict:
        """使用 LLM 抽取實體"""
        prompt = f"""
        從以下文字中抽取結構化資訊，以 JSON 格式輸出：
        - amounts: 所有金額（包含幣別與數字）
        - dates: 所有日期
        - product_names: 產品或設備名稱
        - quantities: 數量
        - project_codes: 專案代號
        - people: 人名
        - companies: 公司名稱
        
        文字：
        {text}
        
        JSON 輸出：
        """
        
        return self.llm.parse_json(prompt)
    
    def align_entities(self, email_entities: Dict, attachment_entities: Dict) -> List[Dict]:
        """找出郵件與附件間的實體對應"""
        alignments = []
        
        # 金額對齊（最常見的關聯線索）
        for email_amount in email_entities.get('amounts', []):
            for att_amount in attachment_entities.get('amounts', []):
                if self._amounts_match(email_amount, att_amount):
                    alignments.append({
                        'type': 'amount_match',
                        'email_value': email_amount['text'],
                        'attachment_value': att_amount['text'],
                        'attachment_location': att_amount['location'],  # 如 "Excel Sheet1 R25C5"
                        'confidence': 0.92
                    })
        
        # 日期對齊
        email_dates = set(email_entities.get('dates', []))
        att_dates = set(attachment_entities.get('dates', []))
        common_dates = email_dates & att_dates
        
        if common_dates:
            alignments.append({
                'type': 'date_match',
                'common_dates': list(common_dates),
                'confidence': 0.85
            })
        
        # 產品名稱對齊（使用語意相似度）
        for email_prod in email_entities.get('product_names', []):
            for att_prod in attachment_entities.get('product_names', []):
                similarity = self._semantic_similarity(email_prod, att_prod)
                if similarity > 0.8:
                    alignments.append({
                        'type': 'product_match',
                        'email_mention': email_prod,
                        'attachment_mention': att_prod,
                        'confidence': similarity
                    })
        
        return alignments
    
    def _amounts_match(self, amount1: Dict, amount2: Dict) -> bool:
        """判斷兩個金額是否相同"""
        # 提取數字部分（忽略格式差異）
        num1 = float(re.sub(r'[^\d.]', '', amount1['normalized']))
        num2 = float(re.sub(r'[^\d.]', '', amount2['normalized']))
        
        # 允許 0.1% 的誤差（處理四捨五入）
        return abs(num1 - num2) / max(num1, num2) < 0.001
```

```python
class SemanticLinkDetector:
    """基於向量相似度的語意關聯"""
    
    def __init__(self, embedding_model='text-embedding-3-large'):
        from openai import OpenAI
        self.client = OpenAI()
        self.model = embedding_model
    
    def compute_embeddings(self, texts: List[str]) -> np.ndarray:
        """批次計算嵌入向量"""
        response = self.client.embeddings.create(
            input=texts,
            model=self.model
        )
        return np.array([item.embedding for item in response.data])
    
    def find_semantic_links(self, email_body: str, attachment_content: str, 
                           chunk_size: int = 500) -> List[Dict]:
        """找出郵件與附件內容的語意關聯區塊"""
        
        # 將附件內容切分為段落
        att_chunks = self._chunk_text(attachment_content, chunk_size)
        
        # 計算嵌入向量
        email_embedding = self.compute_embeddings([email_body])[0]
        chunk_embeddings = self.compute_embeddings(att_chunks)
        
        # 計算相似度
        similarities = cosine_similarity([email_embedding], chunk_embeddings)[0]
        
        # 找出高相似度區塊
        links = []
        for idx, score in enumerate(similarities):
            if score > 0.7:  # 相似度閾值
                links.append({
                    'type': 'semantic',
                    'attachment_chunk': att_chunks[idx],
                    'chunk_index': idx,
                    'similarity_score': float(score),
                    'confidence': float(score)
                })
        
        return sorted(links, key=lambda x: x['similarity_score'], reverse=True)
```

```python
import pandas as pd
from openpyxl import load_workbook

class TableFieldExtractor:
    """Excel/CSV 表格欄位抽取器"""
    
    def extract_from_excel(self, file_path: str, schema: Dict) -> Dict:
        """從 Excel 抽取關鍵欄位"""
        wb = load_workbook(file_path, data_only=True)
        results = {}
        
        for sheet_name in wb.sheetnames:
            ws = wb[sheet_name]
            df = pd.DataFrame(ws.values)
            
            # 識別表頭行
            header_row = self._find_header_row(df)
            if header_row is not None:
                df.columns = df.iloc[header_row]
                df = df.iloc[header_row + 1:].reset_index(drop=True)
            
            # 根據 Schema 抽取欄位
            sheet_data = {}
            for field_name, field_config in schema.items():
                # 使用模糊匹配找出對應欄位
                matched_col = self._fuzzy_match_column(
                    field_config['keywords'], 
                    df.columns
                )
                
                if matched_col:
                    sheet_data[field_name] = {
                        'values': df[matched_col].dropna().tolist(),
                        'column': matched_col,
                        'sheet': sheet_name
                    }
                
                # 特殊處理：總計欄位（通常在最後一列）
                if field_config.get('is_total'):
                    total_value = self._find_total_value(df, matched_col)
                    sheet_data[field_name]['total'] = total_value
            
            results[sheet_name] = sheet_data
        
        return results
    
    def _find_header_row(self, df: pd.DataFrame) -> int:
        """自動偵測表頭行（通常是第一列或第二列）"""
        for idx, row in df.iterrows():
            # 判斷是否為表頭：包含多個文字且少數空值
            if row.notna().sum() > len(row) * 0.7:
                return idx
        return None
    
    def _fuzzy_match_column(self, keywords: List[str], columns: List[str]) -> str:
        """模糊匹配欄位名稱"""
        from difflib import get_close_matches
        
        for keyword in keywords:
            matches = get_close_matches(keyword, columns, n=1, cutoff=0.6)
            if matches:
                return matches[0]
        return None
    
    def _find_total_value(self, df: pd.DataFrame, column: str) -> float:
        """找出總計值（通常在最後幾列，且包含'總計'關鍵字）"""
        for idx in range(len(df) - 1, max(0, len(df) - 5), -1):
            row = df.iloc[idx]
            if any(keyword in str(row.iloc[0]) for keyword in ['總計', '合計', 'Total', 'Sum']):
                try:
                    return float(str(row[column]).replace(',', ''))
                except:
                    pass
        return None
```

```python
class EmailAttachmentProcessor:
    """郵件附件完整處理器"""
    
    def __init__(self):
        self.explicit_detector = ExplicitLinkDetector()
        self.entity_detector = EntityAlignmentDetector(llm_client)
        self.semantic_detector = SemanticLinkDetector()
        self.multimodal_extractor = MultimodalFieldExtractor()
        self.table_extractor = TableFieldExtractor()
    
    def process_email_with_attachments(self, email: Dict, attachments: List[Dict]) -> Dict:
        """處理郵件與附件的完整流程"""
        
        results = {
            'email_id': email['email_id'],
            'attachments_processed': [],
            'cross_references': []
        }
        
        # Step 1: 抽取郵件實體
        email_entities = self.entity_detector.extract_entities(email['body'])
        
        # Step 2: 處理每個附件
        for attachment in attachments:
            att_result = {
                'attachment_id': attachment['attachment_id'],
                'filename': attachment['filename'],
                'extracted_fields': {},
                'links_to_email': []
            }
            
            # 根據檔案類型選擇抽取器
            if attachment['mime_type'] in ['application/pdf', 'image/png', 'image/jpeg']:
                # 使用 GPT-4o 多模態抽取
                att_result['extracted_fields'] = self.multimodal_extractor.extract_from_pdf(
                    attachment['file_path'],
                    self.get_schema_for_filename(attachment['filename'])
                )
            
            elif attachment['mime_type'] in ['application/vnd.ms-excel', 
                                             'application/vnd.openxmlformats-officedocument.spreadsheetml.sheet']:
                # 使用專用 Excel 抽取器
                att_result['extracted_fields'] = self.table_extractor.extract_from_excel(
                    attachment['file_path'],
                    self.get_schema_for_filename(attachment['filename'])
                )
            
            # Step 3: 建立與郵件的關聯
            # 3.1 明確指涉
            explicit_links = self.explicit_detector.detect_references(
                email['body'], 
                [attachment]
            )
            att_result['links_to_email'].extend(explicit_links)
            
            # 3.2 實體對齊
            att_entities = self.entity_detector.extract_entities(
                attachment.get('extracted_text', '')
            )
            entity_alignments = self.entity_detector.align_entities(
                email_entities, 
                att_entities
            )
            att_result['links_to_email'].extend(entity_alignments)
            
            # 3.3 語意關聯
            semantic_links = self.semantic_detector.find_semantic_links(
                email['body'],
                attachment.get('extracted_text', '')
            )
            att_result['links_to_email'].extend(semantic_links[:3])  # 取前 3 個最相關
            
            results['attachments_processed'].append(att_result)
        
        # Step 4: 交叉驗證與一致性檢查
        results['cross_references'] = self._cross_validate(
            email_entities,
            results['attachments_processed']
        )
        
        return results
    
    def _cross_validate(self, email_entities: Dict, att_results: List[Dict]) -> List[Dict]:
        """交叉驗證郵件與附件資訊的一致性"""
        validations = []
        
        # 檢查金額是否一致
        email_amount = email_entities.get('amounts', [{}])[0].get('value')
        for att in att_results:
            att_total = att['extracted_fields'].get('total_amount', {}).get('value')
            if email_amount and att_total:
                if self._normalize_amount(email_amount) == self._normalize_amount(att_total):
                    validations.append({
                        'type': 'amount_consistency_verified',
                        'email_value': email_amount,
                        'attachment_value': att_total,
                        'status': 'consistent'
                    })
                else:
                    validations.append({
                        'type': 'amount_mismatch',
                        'email_value': email_amount,
                        'attachment_value': att_total,
                        'status': 'inconsistent',
                        'severity': 'high'
                    })
        
        return validations
```

- 報價項目清單（品名、型號、數量、單價、金額）
- 小計、稅額、總計金額
- 付款條件、交期、保固期限、有效期限

**位置資訊記錄**：每個欄位記錄在文件中的位置（頁碼、儲存格、座標），方便後續引用。[36][41]

## 階段五：語意關聯建立

### 三層關聯偵測架構

**第一層：明確指涉偵測**[42][43]

使用正規表達式識別郵件中直接提到附件的語句：

- 「請參考附件報價單」
- 「如附件所示」
- 「詳見 Excel 第 3 頁」
- 「報價單編號 Q2025-1001」

**第二層：實體對齊偵測**[7][41]

比對郵件與附件中的結構化資訊：

- **金額對齊**：郵件提到「總計 NT$4,500,000」，附件表格底部出現相同金額
- **日期對齊**：郵件與附件包含相同的截止日期或會議時間
- **產品名稱對齊**：使用語意相似度匹配（「Cisco 9300 交換器」對應「Cisco Catalyst 9300-48P」）
- **專案代號對齊**：郵件與附件都提到「TSMC Fab18」

**第三層：語意向量關聯**[44][42]

計算郵件本文與附件內容的嵌入向量相似度：

- 將附件內容切分為段落（chunk size: 500 字元）
- 計算每個段落與郵件本文的餘弦相似度
- 相似度 > 0.7 的段落標記為高度關聯

### 交叉驗證機制

**一致性檢查**：[2]

- 驗證郵件提到的金額與附件報價單是否一致
- 檢查日期邏輯（如「今天發送的合約」時間是否吻合）
- 偵測矛盾資訊（郵件說 10 台設備，附件列出 12 台）

**不一致警示**：當發現矛盾時，系統標記為高嚴重性事件，提示人工審核。[25]

## 階段六：郵件歷程記錄設計

### 雙時態事件溯源架構

採用 **Bi-temporal Event Sourcing** 追蹤兩個時間維度：[5][45][4]

**業務時間（Valid Time）**：郵件實際發生的時間（如郵件發送時間、附件創建時間）。[5]

**系統時間（Transaction Time）**：資料進入系統的時間（如郵件被讀取、分類被修正的時間）。[5]

**時間旅行查詢**：系統能夠回答「2025 年 9 月 15 日時我們認為客戶的報價是多少？」以及「後來發現實際報價是多少？」。[46][5]

### 資料結構設計

**核心表格**：[47][4][32]

**郵件對話串主表（Threads）**：

- 對話串唯一識別碼
- 正規化主旨
- 第一封與最新一封郵件 ID
- 參與者列表
- 專案標籤、狀態、版本號

**郵件事件流表（Email Events）**：

每個操作記錄為事件，包含：

- 事件類型（EmailReceived、EmailSent、EmailReplied、EmailClassified、ClassificationCorrected、AttachmentVersioned 等）
- 業務時間與系統時間
- 執行動作的用戶
- 事件負載（完整郵件資料 JSON）
- 序號、關聯 ID、因果 ID

**附件版本控制表（Attachment Versions）**：

- 附件邏輯 ID（多個版本共用同一邏輯 ID）
- 版本號（1.0.0、2.0.0、2.1.0）
- 檔案雜湊值（SHA-256，用於去重）
- 前一版本 ID（形成版本鏈）
- 版本變更說明
- 提取的內容與元數據
- 是否為當前版本

**郵件關係圖譜表（Email Relations）**：

- 來源郵件 ID、目標郵件 ID
- 關係類型（replies_to、forwards、references、supersedes）
- 信心分數（AI 判斷的準確度）

**投影視圖表（Projections）**：

快速查詢用的物化視圖，包含郵件當前狀態（主旨、寄件人、分類、附件數量、最後更新時間）。[4][5]

### 版本控制策略

**語義化版本號**：[10][48][9]

- **MAJOR.MINOR.PATCH** 格式
- **MAJOR**：重大變更（報價單 v1 → v2，內容完全不同）
- **MINOR**：功能增強（增加新項目、延長保固期限）
- **PATCH**：錯誤修正（修正錯別字、數字錯誤）

**自動版本偵測**：[49][9]

1. **檔名相似度比對**：「報價單v1.0.pdf」vs「報價單v2.0.pdf」
2. **內容雜湊比對**：相同雜湊值不建立新版本（去重）
3. **語意相似度分析**：使用嵌入向量判斷是否為新版本
4. **變更內容記錄**：自動提取版本間差異（價格調整、項目增減）

### 對話串時間軸結構

每個對話串包含完整的里程碑記錄：

- **初始詢問**：客戶首次聯繫，提出需求
- **報價提供**：業務回覆初步報價（附件：報價單 v1.0）
- **議價協商**：客戶要求降價或修改條件
- **報價修訂**：業務調整報價（附件：報價單 v2.0，記錄差異）
- **最終確認**：客戶同意並發送採購單

每個里程碑記錄：執行者、時間戳記、描述、附件、關鍵實體、變更內容。[2]

## 階段七：增量學習與自我進化

### 冷啟動階段（前 100-500 封郵件）

**初期策略**：[6]

- 使用預訓練模型進行基礎分類（如 GPT-4o、Gemini）
- 人工審核高不確定性預測（信心分數 < 0.6）
- 建立初始知識圖譜與關係網絡
- 收集使用者反饋作為訓練資料

### 主動學習階段（500-5000 封郵件）

**不確定性採樣**：[50][51]

系統自動標記「難以判斷」的郵件，請求人工確認：

- 分類信心分數低於閾值（< 0.7）
- 多個類別分數接近
- 新出現的寄件人或主題模式

**概念漂移偵測（Concept Drift Detection）**：[3]

當郵件模式改變時（如新業務線、新客戶類型、組織改組）自動觸發重新訓練：[3]

- 監控分類準確率趨勢
- 偵測新詞彙或術語出現頻率
- 追蹤人際網絡變化

**滑動視窗機制**：[3]

保留最近 3-6 個月的郵件作為「新鮮知識」，舊資料權重降低，確保模型適應最新業務狀態。[3]

### 自主優化階段（5000 封以上）

**特徵空間動態擴展**：[6]

使用增量 PCA（Principal Component Analysis）自動發現新的郵件模式，無需重新訓練全部資料。[6]

**關係強度更新**：

根據互動頻率動態調整人際網路權重，反映當前組織狀態。[6]

**分類器自適應**：[52][6]

每週或每月自動更新模型參數，使用 SGDClassifier（Stochastic Gradient Descent）等增量學習演算法。[52]

### 反饋迴路機制

**隱式反饋**：[51]

- 追蹤使用者行為（哪些郵件被點擊、附件被下載、回覆速度）
- 記錄查詢成功率（使用者是否找到所需資訊）
- 分析停留時間與互動深度

**顯式反饋**：[53][50]

- 提供「標記錯誤」按鈕，使用者糾正分類錯誤
- 收集「有用」「無用」評分
- 定期邀請關鍵使用者進行品質審核

**A/B 測試**：[51]

對比不同分類策略的效果，自動選擇最佳方案（如比較規則式分類 vs. AI 分類的準確率）。[51]

### 進化指標追蹤

**關鍵績效指標（KPI）**：[50][3]

- **分類準確率變化**：初期 70% → 3 個月後 85% → 6 個月後 > 90%
- **人工干預頻率下降**：從每天 50 次 → 每週 5 次
- **關聯抽取精度**：附件與郵件對應的 F1 score
- **檢索召回率**：使用者查詢能找到正確附件的比例
- **平均回應時間**：系統回答查詢的速度

## 階段八：向量化與索引建立

### 嵌入模型選擇

**商用方案**：[24]

- **OpenAI text-embedding-3-large**：3072 維，支援多語言
- **Google Gemini Embedding**：與 Gemini API 無縫整合

**開源替代**：[24]

- **BGE-M3**：多語言支援，適合中英混合文本
- **Stella-v5**：最新開源嵌入模型，效能接近商用方案

### 分塊策略

**動態切分**：[31][24]

- **短郵件**（< 500 字）：整封郵件為一個 chunk
- **中郵件**（500-2000 字）：依段落切分，保留 50-100 字重疊區
- **長郵件**（> 2000 字）：使用滑動視窗，確保語意連貫性

**保留元數據**：每個 chunk 記錄來源郵件 ID、對話串 ID、位置索引、時間戳記、寄件人。[24]

### 向量資料庫選擇

**輕量級方案**：

- **SQLite-vec**：適合單機部署，資料量 < 100 萬筆
- **Chroma**：易於整合，適合快速原型開發

**進階方案**：[24]

- **Qdrant**：高效能，支援混合檢索（向量 + 關鍵字）
- **Milvus**：分散式架構，支援億級資料規模
- **Pinecone**：全託管雲端服務，零維運負擔

### 混合索引策略

**結合全文檢索與向量搜尋**：[24]

- **BM25（全文檢索）**：精確匹配關鍵字（如訂單號、產品型號）
- **向量搜尋**：語意相似度匹配（如「網路設備報價」匹配到「Cisco 交換器詢價」）
- **重排序（Reranker）**：使用 Cross-Encoder 模型對候選結果重新排序

**評分融合**：

結合兩種檢索結果的分數，使用加權平均或 Reciprocal Rank Fusion（RRF）演算法。[24]

## 階段九：RAG 系統整合

### 檢索增強生成架構

**查詢流程**：[54][24]

1. **使用者查詢**：「上次 TSMC 專案的報價多少？」
2. **查詢理解**：抽取關鍵實體（TSMC、報價、金額）
3. **混合檢索**：同時執行向量搜尋與關鍵字搜尋
4. **結果重排序**：使用 Reranker 提升相關性
5. **上下文增強**：為每個檢索結果加入元數據描述[54]
6. **LLM 生成**：將檢索結果與查詢傳送給 LLM 生成答案
7. **引用標註**：標記答案來源（郵件連結、附件頁碼）

### 上下文增強技術

**Contextual Retrieval**：[54]

為每個 chunk 加入上下文描述，避免孤立片段造成的理解偏差。[54]

範例：

- **原始 chunk**：「總金額 NT$4,500,000」
- **增強後**：「這是 2025 年 10 月 TSMC Fab18 專案的 Cisco 網路設備報價單，總金額 NT$4,500,000」

### Prompt 模板設計

**針對不同查詢類型建立範本**：[24]

**歷史查詢**：「請根據以下郵件記錄，回答關於 {專案名稱} 的 {詢問內容}...」

**版本比對**：「比較以下兩個版本的報價單差異...」

**決策追溯**：「根據對話串，說明 {決策事項} 的演變過程...」

### 測試與驗證

建立 20-50 個典型問題測試集：[24]

- 「去年與某客戶的合約金額是多少？」
- 「哪些專案提到 Cisco 9300 交換器？」
- 「為什麼報價從 500 萬降到 450 萬？」
- 「上次會議的參與者有誰？」

**評估指標**：

- **召回率（Recall）**：檢索到的相關文件比例
- **精確率（Precision）**：檢索結果中相關文件比例
- **MRR（Mean Reciprocal Rank）**：正確答案排名的倒數平均值
- **使用者滿意度**：實際使用者評分

## 階段十：品質控管與驗證

### 資料品質檢查

**完整性驗證**：[25][2]

- 必填欄位檢查（如郵件必須有寄件人、時間戳記）
- 附件完整性確認（檔案是否損毀、大小是否異常）
- 關聯完整性（引用的郵件 ID 是否存在）

**準確性檢核**：[2]

- AI 提取結果的信心分數篩選（threshold > 0.7）
- 交叉驗證郵件與附件資訊一致性
- 偵測邏輯矛盾（如日期倒序、金額不合理）

**異常偵測**：[2]

- 過長郵件（> 10,000 字，可能為系統自動通知）
- 過短郵件（< 10 字，可能為垃圾郵件）
- 亂碼內容（編碼錯誤）
- 重複 ID（資料匯入錯誤）

### 人工審核介面（HITL）

**Human-In-The-Loop 設計**：[25]

針對低信心分數或異常資料，系統標記供人工確認：

- 提供清晰的審核介面（並排顯示郵件與附件）
- 支援快速批次操作（核准、拒絕、修正）
- 記錄審核歷史，作為訓練資料

**審核優先級**：

- **高優先**：涉及金額、合約、關鍵決策的郵件
- **中優先**：新客戶、新業務類型
- **低優先**：一般通知、內部溝通

```python
# version_tracker.py
import psycopg2
from danswer_client import DanswerAPI

class VersionTracker:
    """監聽 Danswer 索引事件，追蹤附件版本"""
    
    def __init__(self):
        self.danswer = DanswerAPI()
        self.db = psycopg2.connect("dbname=email_versions")
    
    def on_document_indexed(self, doc):
        """當 Danswer 索引新文件時觸發"""
        if self.is_attachment(doc):
            self.check_version(doc)
    
    def check_version(self, doc):
        """檢查是否為現有附件的新版本"""
        similar_docs = self.danswer.search(
            query=doc.filename,
            filters={"file_type": doc.file_type}
        )
        
        if similar_docs:
            # 判定為新版本，記錄到資料庫
            self.save_version(doc, previous=similar_docs[0])
```
