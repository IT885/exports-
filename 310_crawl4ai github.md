# Crawl4AI GitHub 專案介紹

Crawl4AI 是 GitHub 上排名第一的趨勢開源專案，由活躍社區維護，專為大型語言模型 (LLM)、AI 代理和數據管道提供極速且 AI 友好的網頁爬蟲和抓取功能。[1][2]

## 核心功能

**生成乾淨的 Markdown**：完美適配 RAG 管道或直接輸入到 LLM[2]

**結構化提取**：使用 CSS、XPath 或基於 LLM 的提取來解析重複模式[2]

**高級瀏覽器控制**：提供鉤子、代理、隱身模式、會話重用等細粒度控制[2]

**高性能**：支援並行爬蟲、基於區塊的提取，適合實時使用場景[2]

**開源免費**：無強制 API 密鑰，無付費牆-所有人都能訪問自己的數據[2]

## 快速入門示例

以下是展示 Crawl4AI 異步功能使用的簡單代碼示例：

```python
import asyncio
from crawl4ai import AsyncWebCrawler

async def main():
    # 創建 AsyncWebCrawler 實例
    async with AsyncWebCrawler() as crawler:
        # 在 URL 上運行爬蟲
        result = await crawler.arun(url="https://crawl4ai.com")
        # 打印提取的內容
        print(result.markdown)

# 運行異步主函數
asyncio.run(main())
```


## 相關 GitHub 專案

Crawl4AI 生態系統中還有多個相關專案：

**Crawl4AI-MCP**：智能網頁爬蟲和 AI 驅動內容分析服務器，結合了有針對性的網頁抓取與 Claude AI 處理功能[3]

**BjornMelin/crawl4ai-mcp-server**：高性能 MCP 服務器，使 AI 助手能夠訪問網頁抓取、爬行和深度研究功能，比 FireCrawl 更快更高效[4]

**weidwonder/crawl4ai-mcp-server**：基於 MCP (Model Context Protocol) 的中文版智能信息獲取服務器，為 AI 助手系統提供強大的搜索能力和面向 LLM 優化的網頁內容理解功能[6]

## 應用實例

**AI Web Scraper**：使用 Crawl4AI 構建的 AI 驅動網頁抓取工具，可自動提取當地企業（如牙醫、餐廳等）的名稱、地址、電話號碼等信息，通過 GPT-4o、Claude 和 DeepSeek 等 LLM 智能處理數據並保存為 CSV 文件，適用於潛在客戶開發或分析[5]

## 核心理念

Crawl4AI 專注於數據民主化，提供免費、透明且高度可配置的工具。同時，它特別針對 LLM 友好設計，提供最少處理的、結構良好的文本、圖像和元數據，使 AI 模型能夠輕鬆使用這些內容。[2]

# 如何使用Crawl4AI生態系統中的MCP服務器

Crawl4AI生態系統提供了強大的網頁爬蟲工具，特別是通過MCP（Model Context Protocol）服務器實現與AI助手的整合。以下是使用這些服務器的白話指南。

## 基本設置步驟

**安裝Crawl4AI基礎工具**：
- 使用Python的pip安裝：`pip install crawl4ai`[3]
- 運行設置命令確保依賴項配置正確：`crawl4ai-setup`[3]
- 如遇問題，使用診斷工具：`crawl4ai-doctor`[3]

**配置MCP服務器**：
1. 克隆相關倉庫：`git clone https://github.com/weidwonder/crawl4ai-mcp-server.git`（以weidwonder版本為例）[6][3]
2. 安裝必要依賴：`pip install -r requirements.txt`[3]
3. 配置API金鑰（如需使用Google搜索功能）[2][3]
4. 啟動服務器：`python app.py`[3]

## weidwonder/crawl4ai-mcp-server功能使用

這是基於MCP的中文版智能信息獲取服務器，提供兩個核心功能：

**搜索功能（search）**：
- 支持DuckDuckGo搜索（默認，無需API密鑰）及Google搜索（需API密鑰）[2]
- 使用方法：發送包含查詢字符串、結果數量和搜索引擎選擇的請求[2]
- 參數設置：
  - query：搜索查詢字符串
  - num_results：返回結果數量（默認10）
  - engine：選擇"duckduckgo"、"google"或"all"[2]

**網頁內容理解（read_url）**：
- 智能提取網頁內容並轉換為LLM友好格式[2]
- 多種輸出格式：
  - markdown_with_citations：包含內聯引用的Markdown（默認）
  - fit_markdown：經過LLM優化的精簡內容
  - raw_markdown：基礎HTML→Markdown轉換[2]

**進階應用**：

這樣設置後，您的AI助手就能夠直接從網頁獲取信息，進行分析和處理，大大提升AI與網路信息交互的能力。[2][4]

# Crawl4AI如何處理資訊與網頁搜尋功能

Crawl4AI本身並非搜尋引擎，而是一個專為AI應用優化的網頁爬蟲和數據提取工具。讓我來解釋它的核心工作機制：

## 基本架構與數據處理

Crawl4AI是一個開源Python庫，基於**異步架構**設計，使用Python的`asyncio`框架和`Playwright`作為底層瀏覽器自動化引擎。它不含內建完整的AI模型，而是提供多種數據提取機制：[6]

- **啟發式演算法**：使用BM25算法和其他智能演算法來過濾和優化網頁內容[5]
- **結構化提取**：支援CSS/XPath選擇器進行精確數據提取[1][5]
- **LLM驅動的提取**：通過與外部LLM整合實現智能提取[3][4]

## 關於搜尋功能

Crawl4AI如何進行"網頁搜尋"主要有兩種方式：

1. **通過MCP服務器整合搜尋引擎**：
   - 某些MCP服務器實現（如weidwonder/crawl4ai-mcp-server）提供了搜尋引擎整合功能
   - 支持DuckDuckGo搜尋（默認，無需API密鑰）及Google搜尋（需API密鑰）[2]
   - 可通過API發送搜尋請求並獲取結果

2. **網站爬取與數據發掘**：
   - 支持深度爬取（deep crawl）功能，使用BFS等策略爬取大量頁面[5]
   - 提供完整的鏈接提取功能，包括內部和外部鏈接[5]
   - 可設置最大頁面數限制，如：`crwl https://docs.crawl4ai.com --deep-crawl bfs --max-pages 10`[5]

## 與LLM的整合方式

Crawl4AI需要外部LLM來實現高級功能：

- 需要設置外部API密鑰：`export OPENAI_API_KEY='your_api_key_here'`[4]
- 可與DeepSeek等開源模型整合處理複雜數據[3]
- 透過命令行參數傳遞特定問題給LLM：`crwl https://www.example.com/products -q "Extract all product prices"`[5]

總結來說，Crawl4AI本身不包含完整的搜索引擎或大型語言模型，而是作為一個高效的中間層，連接網頁內容與AI模型，專注於提供乾淨、結構化、AI友好的數據。它的核心價值在於將混亂的網頁內容轉換為LLM可以高效處理的格式。

# Crawl4AI與本地Qwen3-1.7B模型整合說明

您的理解基本正確。Crawl4AI的工作流程與本地模型整合方式如下：

## 搜尋引擎整合

- 確實，Crawl4AI的MCP服務器實現中，有些搜尋引擎需要API金鑰（如Google），有些則不需要（如DuckDuckGo）[2]
- 它的確會像搜尋引擎一樣返回"第一層"結果，但其核心功能更在於深入爬取和分析網頁內容[2]

## 與本地Qwen3-1.7B整合

是的，您可以使用本地部署的Qwen3-1.7B模型與Crawl4AI配合：

1. **LM Studio完全支援Qwen3模型**：
   - 官方文檔明確表示："對於本地使用，LMStudio、Ollama、MLX-LM、llama.cpp和KTransformers等應用程序也已支援Qwen3"[1]

2. **通過LM Studio開啟API**：
   - LM Studio可以提供OpenAI兼容的API端點
   - 在LM Studio中載入Qwen3-1.7B模型後，可以在設置中啟用"Local Server"，它會開啟一個與OpenAI兼容的API端點

3. **配置Crawl4AI使用本地模型**：
   - 在Crawl4AI代碼中，將LLMConfig設置為指向您的本地LM Studio API：
     ```python
     llm_config = LLMConfig(
         provider="openai", 
         api_token="不需要真實token", 
         base_url="http://localhost:1234/v1"  # LM Studio的默認端口
     )
     ```

4. **支援思考模式**：
   - Qwen3-1.7B支援"思考模式"和"非思考模式"的無縫切換，可以處理複雜邏輯推理和一般對話[4][1]
   - 這對於網頁內容的深度分析特別有用

Qwen3-1.7B在本地運行時，雖然比較輕量（1.7B參數），但仍提供了不錯的多語言處理能力和32K上下文窗口，對於處理Crawl4AI抓取的內容足夠勝任。[5][1]

這種本地部署方式還能避免API費用和數據隱私問題，適合進行實驗性質的網頁內容處理項目。

# 知識庫是RAG的重要組成部分，而非RAG本身

知識庫只是RAG（檢索增強生成）系統中的一個重要組件，而非完整的RAG系統。

## 知識庫在RAG中的角色

知識庫是RAG架構的基礎元素，其作用主要是：
- 儲存企業內部資料與數據，作為檢索的來源[1]
- 透過向量資料庫形式存放經過處理的文字、圖片、聲音、影片等非結構化資料[2][4]
- 作為RAG模型運作流程的第一步，即「知識庫構建」階段[5]

## RAG的完整架構

RAG不僅僅是知識庫，而是一個完整的處理框架，包含以下關鍵組件：

1. **索引（Index）**：將企業資料預處理並放進知識庫[1]
2. **檢索（Retriever）**：當使用者提問時，從知識庫中找出相關資訊[3][1]
3. **生成（Generator）**：將檢索到的資料提供給LLM生成答案[3][1]

RAG的核心概念是「在生成回應前，從外部資料庫中檢索相關資訊，再與原本要詢問LLM的問題結合，一併提供給LLM」，這個過程涉及多個技術環節，知識庫只是其中一部分。[4][2]

如IBM語言技術總監Luis Lastras所述，RAG系統要求模型從資料庫中檢索內容，而不僅依靠記憶回答問題，這說明知識庫是RAG的重要組成，而非RAG本身。[4]

## 多模態RAG (MRAG)

多模態RAG是一種能夠處理文本、圖像和視頻等多種媒體類型的增強技術。根據最新研究，MRAG已經發展到3.0版本：

- **MRAG 2.0**：在檢索和生成過程中真正保留了多模態數據，使用多模態檢索和多模態大型語言模型(MLLMs)[9]
- **MRAG 3.0**：整合了文檔截圖以減少信息丟失，引入多模態搜索規劃模塊，支持多模態輸出[9]

- **增強生成**：使用LLM處理並整合數據，生成上下文準確的回應[2]

## 模塊化RAG

一種更靈活的RAG實現方式，將RAG系統拆解為可重組的模塊：

- **三層架構設計**：L1 Module（核心過程）、L2 Sub-module（功能細化）、L3 Operator（具體功能實現）[4]
- **靈活重組**：允許研究人員在統一框架內複製現有RAG方法或開發新算法[4]

突破傳統RAG限制的新技術，直接將搜索引擎功能嵌入到LLM中：

- 簡化開發流程，提高模型在複雜任務中的表現[8]

## 與非結構化數據庫的整合

現代RAG系統越來越多地整合非結構化數據庫技術：

- **Google BigQuery object table**：為存儲在Cloud Storage中的非結構化數據提供結構化記錄介面[1]

- **LLM Ware**：建立企業級RAG管道的框架，提供50多個微調小型模型[3]
- **RagFlow**：專注於深度文件理解的RAG引擎，支持基於引文的問答[3]

RAG技術的快速發展確實可能改變相關工程師的工作性質：

- **標準化趨勢**：底層軟體模式日益標準化，使RAG建置與部署變得更加容易[6]
- **針對RAG最佳化的LLM**：如Perplexity AI已針對各種RAG應用進行微調[6]
- **自動化程度提高**：FlashRAG等高效模塊化工具包使研究人員能更輕鬆開發RAG算法[4]
