# 技術問題分析報告（長官版）

## 問題根本原因確認

### **故障設備識別**
- **問題設備**：HZ-13516 及 HZ-13517 兩台網路交換器
- **錯誤配置**：兩台設備間的網路連接埠設定錯誤，造成**網路迴圈**

### **具體問題描述**
**簡單說明**：就像水管接錯了方向，水一直在管子裡打轉，最後把整個系統都塞住了。

**技術細節**：
- HZ-13517 的網路埠被錯誤設定為一般存取埠
- 這個埠實際上應該連接到另一台設備的主幹埠
- 錯誤連接導致網路流量在設備間無限循環
- 觸發**廣播風暴保護機制**，但仍造成產線設備受影響

## 為什麼防護機制沒有作用？

### **REP保護機制局限性**
**問題核心**：現有的REP（Resilient Ethernet Protocol）保護機制有設計盲點

**具體限制**：
- REP只能控制**同一網段內**的連接埠
- 對於**未加入REP保護的一般存取埠**無法管控
- 無法監控其他網路連線的迴圈狀況
- 因此無法阻止這次的錯誤連線造成的問題

## Cisco原廠解決方案

### **立即改善措施**
1. **嚴格配線檢核**
   - REP環內的主幹埠必須正確對接其他REP主幹埠
   - 安裝前必須二次確認正確位置
   - 避免人為接線錯誤

2. **預防機制強化**
   - 所有參與REP的設備都必須正確設定為主幹模式
   - 並允許相關VLAN標籤通過

## 業務影響評估

### **客戶影響**
- ✅ **問題根因已確認**：接線錯誤導致
- ✅ **解決方案明確**：重新正確配線
- ⚠️ **預防措施**：需要建立更嚴格的安裝檢核程序

### **後續行動**
1. **立即執行**：現場重新配線並測試
2. **流程改善**：建立配線雙重檢核機制
3. **人員訓練**：加強工程師REP協定理解

# F22P1_3F ZCU11 Broadcast Storm 簡報內容

## 目錄（Agenda）

1. Root Cause 分析
2. Cisco TAC 確認事項
3. 建議作為／避免再發生

## 1. Root Cause 分析

- **事件發生時間**：2025/6/17 17:05 起
- **異常現象**：ZCU11 RING 發生 broadcast storm，產線網路異常
- **現場 LOG 重點**：
  - Corhub 端於 17:06 觀察到 interface e1/27（RING ZCU11）storm control action shutdown
  - 觸發 broadcast & multicast 10Mb 閾值，系統自動保護
  - LOG 時間有約 30 秒誤差，因 OS NTP 校時 bug，實際為同一時間
- **LOOP 問題來源**：
  - HZ-13517 的 Gi1/4 與 Gi1/10 同時收到相同流量，證實 loop 起點
  - LOOP 發生後，broadcast 流量擴散至其他 ZCU RING
  - ZCU11 uplink 達到閾值後由 Corhub01/02 shutdown，異常才被阻斷
  - HZ-13517 的 loop 行為間歇持續至 18:55 才完全停止
- **異常連線分析**：
  - Gi1/3 應接 HZ-13516，實際未連接
  - Gi1/10 應為空 port，卻顯示 connected
  - REP topology 未形成正常 RING
  - MAC address 查詢發現 Gi1/10 對端為 HZ-13516，確認網路線接錯 port
  - 可能原因：Gi1/3 與 Gi1/10 在面板上排列相近，易誤插

## 2. Cisco TAC 確認事項

- **原廠技術回覆重點**：
  - HZ-13516 Gi1/4 與 HZ-13517 Gi1/3 正確設定為同一 REP segment 的 trunk port
  - HZ-13517 Gi1/10 被設定為 access port，卻誤接至 Gi1/3，產生非預期 loop path
  - VLAN 21 流量在 Gi1/10 → Gi1/3 → Gi1/4 間循環，形成 Layer 2 迴圈
  - LOG 顯示 MAC flapping between Gi1/4 and Gi1/10
- **REP 機制說明**：
  - REP 只能管理同一 segment 的 trunk port
  - 未設定 REP 的 access port（如 Gi1/10）不會交換控制訊息，無法監控其轉發狀況
  - 因此無法封鎖這條錯誤連線造成的 loop
  - REP protocol 要求所有參與埠皆需設定為 trunk，允許對應 VLAN 標籤

## 3. 建議作為／避免再發生

- **持續進行 LAB 驗證**  
  在實驗室模擬各種情境，針對問題進行全面驗證

- **加強現場人力巡檢**  
  安排工程師到場，定期檢查設備連線與設定，及早發現異常

- **導入自動化異常偵測**  
  建置自動化監控機制，第一時間偵測並通報網路異常，提升反應速度

- **思科原廠技術改善建議**  
  - 所有參與 REP 環的連接埠應正確設定為 trunk port，避免 access port 誤接
  - 安裝前後需雙重檢查設定
  - 強化自動化監控與教育訓練，確保網路架構穩定、降低人為疏失風險

## 4. LAB 驗證內容與技術分析

  1. 將 Gi1/3 與 Gi1/10 錯誤連接
  2. 確認形成 loop，產生 broadcast storm

  - 只要 access port 誤接進 REP 環，且未被 REP 管理，就會產生 loop
