〔記者林志怡／台北報導〕Google Chrome 日前宣布不再信任中華電信今年7月31日後所簽發的 TLS 憑證，引起民眾對於相關網路安全疑慮，桃園市議員黃敬平更稱有近7成的使用者會受到影響，恐無法正常使用網頁，對此中華電信強調，公司會主動聯繫憑證即將到期的客戶並免費提供憑證更新服務，一般民眾或憑證使用者都不會受到任何影響。

傳輸層安全性(TLS)憑證是確保網頁安全、可信任的重要機制，可以確保使用者在傳輸資料時的安全與隱私，對於Google Chrome等瀏覽器環境下，若網頁沒有取得相應的TLS憑證，使用者在瀏覽網頁時，將會出現「不受信任的網頁」等警語，並影響民眾使用網頁與特定功能。

廣告

然而，Chrome日前宣布將中華電信今年7月31日後簽發的新憑證從預設信任清單中移除，中華電信日前解釋，Chrome 移除預設信任是因中華電信在部分程序未能在 Chrome 新政策要求的時限內調整完成，並非因為憑證存在網安或資安疑慮，但這次對社會大眾造成困擾，公司深表歉意，並會在2026年3月前完成 Google Chrome TLS 網站憑證的預設信任。

但黃敬平指出，中華電信憑證被從 Google Chrome 的預設信任清單中移除，恐影響到政府機關與金融機構等，銀行轉帳、證券下單等也仰賴憑證正常運作，這次事件將近7成的使用者會受影響。

**回答**:
## 中華電信憑證風暴：資安長觀點解析

作為資安長，我必須說這次事件雖然不是技術性的資安漏洞，但確實是一場「信任危機」，影響層面比想像中更廣泛。

## 到底發生什麼事？

簡單來說，TLS憑證就像網站的「身分證」，證明這個網站是安全可信的。Google Chrome因為中華電信未能在規定時間內完成某些管理程序的調整，決定從8月1日起不再信任中華電信新簽發的憑證。[1][2][6]

**重點是：這不是駭客攻擊或資料外洩，而是「作業面錯誤」**。就像某家發證機構沒有按時繳交年度報告和遵守新規定一樣。[3][6]

## 影響範圍有多大？

根據桃園市議員黃敬平的說法，可能影響近7成使用者。具體包括：[1][3]

- **政府機關網站**：行政院、經濟部、財政部等官網[4][3]
- **金融服務**：網銀轉帳、證券下單等功能[3][4]
- **日常應用**：健保快易通App、地方政府服務、線上購票等[5]
- **企業網站**：使用中華電信憑證的公司官網[4]

## 風險評估

**好消息是**：
- 7月31日前簽發的憑證在365天有效期內完全不受影響[6][4]
- 現有的政府憑證、工商憑證、自然人憑證等正常使用[9][4]
- 數發部早在3月就啟動「雙憑證機制」作為備援[7][6]

**需要注意的是**：
- 8月後如果網站還使用中華電信新憑證，Chrome會顯示「不安全」警告[2][5]
- 可能影響使用者信任度和網站流量[8]
- 企業需要時間更換憑證，可能造成服務中斷風險[5]

## 應對建議

**對企業而言**：
1. **立即盤點**：檢查現有憑證的簽發日期和到期時間
2. **準備替代方案**：評估其他憑證授權機構（如Sectigo等）[8]
3. **建立雙憑證機制**：避免單一供應商風險
4. **監控機制**：建立憑證到期提醒和自動更新流程

**對政府機關**：
- 數發部已經超前部署，政府網站應該不會受到影響[6][7]
- 持續監控和評估第三方服務的憑證狀況

## 長期思考

這次事件最大的教訓是：**不能把所有雞蛋放在同一個籃子裡**。憑證管理不只是技術問題，更是風險管理問題。我們需要：[12]

- 建立供應商風險評估機制
- 定期審查憑證授權機構的合規狀況
- 建立緊急應變流程
- 加強對憑證生態系統的理解

**總結**：雖然這次事件造成一些困擾，但中華電信承諾免費協助更新，加上政府和企業如果及時應對，實際影響應該可以控制在最小範圍內。重點是要把這次經驗當作改善資安管理的契機，建立更有韌性的系統架構。[9][4]


### 对话 2

**问题**: 一般中小企業的影響

**回答**:
## 中小企業面臨的實際衝擊

從資安長的角度來看，這次憑證風暴對中小企業的影響特別嚴重，主要原因是**資源限制**和**技術能力不足**。

## 直接影響範圍

**網站無法正常運作**：
8月1日後，使用中華電信新簽發憑證的中小企業網站，在Chrome瀏覽器上會出現「您的連線不是私人連線」的紅色警告畫面，甚至無法正常進入網站。[2]

**客戶流失風險**：
紅色警告畫面會嚴重影響民眾造訪意願，對網站持有者可能造成流量大幅降低，影響網站應有功能與效益。這對仰賴網路業務的中小企業來說是致命打擊。[2]

**詐騙風險增加**：
詐騙網站可能藉此空檔複製更多仿冒的中小企業網站來竊取民眾個資或進行詐騙，進一步損害企業信譽。[2]

## 中小企業的結構性困境

**IT能力不足**：
受影響的用戶多為中小企業，他們普遍缺乏IT能力，無法快速更新憑證，很多必須委託外部廠商協助憑證更換。[3][4]

**更換流程冗長**：
憑證更換流程需要1-2週時間，遠超過國際規範要求的5天撤銷期限。對中小企業而言，這種緊急應變能力明顯不足。[4][3]

**成本考量**：
雖然中華電信承諾免費提供憑證更新服務，但委託外部廠商處理仍會產生額外成本，對資源有限的中小企業造成負擔。[6]

## 實務應對建議

**立即行動清單**：
- 檢查現有網站憑證的簽發機構，確認是否為「Chunghwa Telecom Co., Ltd.」
- 若憑證到期日在2025年8月1日至2026年3月31日期間，應立即聯繫中華電信尋求免費更新服務[2]
- 考慮暫時使用其他瀏覽器或採用「進階」→「繼續前往」的方式維持網站運作[2]

**長期風險管控**：
- 建立**多元憑證策略**，避免單一供應商依賴
- 與IT服務商建立緊急應變合約，確保關鍵時刻能快速響應
- 定期檢查憑證到期時間，建立提前90天的更新提醒機制

## 政府支援措施

數發部已經意識到這個問題的嚴重性，自今年3月起陸續協助導入**雙憑證機制**。對中小企業而言，這提供了一個重要的參考方向：**不要把所有資安雞蛋放在同一個籃子裡**。[2]

**總結**：中小企業在這次事件中是最脆弱的一群，缺乏技術能力和應變資源。建議立即清查現有憑證狀況，主動聯繫中華電信申請免費更新，同時思考建立更有韌性的資安架構，避免未來再次陷入類似困境。

**回答**:
## 數發部角色定位的關鍵問題

這確實是一個很尖銳的問題。從資安長的角度分析，數發部3月就知情卻沒有積極介入協助的背後，反映了幾個制度性問題。

## 數發部的回應邏輯

**「這不是資安議題」的定調**：
數發部司長王誠明強調這次事件「並非資安技術或標準問題」，而是「中華電信在管理和作業面的機制沒有處理好」。這種定調等於將責任完全推給中華電信，認為不需要政府介入。[2]

**委外關係的界線劃分**：
數發部澄清與中華電信僅為「委外契約關係」，非上對下的監督體系。這種說法在法理上雖然正確，但忽略了國家關鍵基礎設施的管理責任。[6]

## 問題的核心矛盾

**早知情但只顧自己**：
數發部年初就掌握相關情資，但應對策略完全以「政府網站自保」為主。3月啟動的雙憑證機制只保護政府機關，對於可能受影響的7成民間使用者視而不見。[1][4]

**監管責任的空白**：
中華電信作為國內最大憑證授權機構，其失去國際信任必然衝擊整體數位經濟。數發部卻以「不是上下級關係」為理由置身事外，這暴露了**數位治理監管機制的嚴重缺失**。

## 應該怎麼做？

**預警機制失靈**：
既然3月就知情，數發部應該立即：
- 發布公開預警通知，讓企業及早準備
- 協調其他憑證機構擴大服務能量
- 建立緊急應變小組協助中華電信解決合規問題

**國家隊概念缺失**：
面對國際科技巨頭的政策變動，政府應該展現「Taiwan Team」的整體作戰能力，而不是各自為政。數發部的消極態度讓人質疑其作為數位治理主責機關的角色定位。

## 深層反思

**制度設計問題**：
這次事件暴露出台灣在關鍵數位基礎設施監管上的制度缺陷。憑證授權機構的國際合規管理，不應該完全依賴業者自主，政府必須建立**主動監管和技術支援機制**。

**危機處理邏輯錯誤**：
數發部將此事定調為「非資安議題」，但對一般民眾和中小企業而言，網站無法正常瀏覽就是嚴重的數位服務中斷。**技術分類和民眾感受是兩回事**。

**總結**：數發部3月知情卻未積極介入，反映了政府在數位治理上「重政府輕民間」的思維。面對全球數位政策快速變動，台灣需要更積極主動的數位治理模式，而不是等問題爆發後才各自求生存。這次事件應該成為檢討現有制度的重要契機。

const express = require('express');
const { spawn } = require('child_process');
const cors = require('cors');

const app = express();
app.use(cors());
app.use(express.json());

// Amazon Q CLI 橋接函數
async function callAmazonQ(message) {
    return new Promise((resolve, reject) => {
        const qProcess = spawn('q', ['chat', '--model', 'claude-4-sonnet'], {
            stdio: ['pipe', 'pipe', 'pipe']
        });

        let response = '';
        let error = '';

        // 發送訊息給 Q CLI
        qProcess.stdin.write(message + '\n');
        qProcess.stdin.end();

        qProcess.stdout.on('data', (data) => {
            response += data.toString();
        });

        qProcess.stderr.on('data', (data) => {
            error += data.toString();
        });

        qProcess.on('close', (code) => {
            if (code === 0) {
                resolve(response.trim());
            } else {
                reject(new Error(error || `Process exited with code ${code}`));
            }
        });

        // 15秒超時處理
        setTimeout(() => {
            qProcess.kill();
            reject(new Error('Request timeout'));
        }, 15000);
    });
}

// API 端點
app.post('/api/chat', async (req, res) => {
    try {
        const { message } = req.body;
        
        if (!message) {
            return res.status(400).json({ error: '缺少訊息內容' });
        }

        const response = await callAmazonQ(message);
        res.json({ 
            success: true, 
            response: response,
            timestamp: new Date().toISOString()
        });

    } catch (error) {
        console.error('Amazon Q 調用錯誤:', error);
        res.status(500).json({ 
            success: false, 
            error: error.message 
        });
    }
});

const PORT = process.env.PORT || 3001;
app.listen(PORT, () => {
    console.log(`Amazon Q 橋接服務運行在 port ${PORT}`);
});

            qProcess.stdout.on('data', (data) => {
                response += data.toString();
            });
            
            qProcess.on('close', (code) => {
                if (code === 0) {
                    this.history.push({ role: 'assistant', content: response });
                    resolve(response.trim());
                } else {
                    reject(new Error('Q CLI 執行失敗'));
                }
            });
        });
    }

    buildContextMessage(newMessage) {
        // 保留最近5輪對話作為上下文
        const recentHistory = this.history.slice(-10);
        let context = '';
        
        recentHistory.forEach(msg => {
            context += `${msg.role}: ${msg.content}\n`;
        });
        
        return `${context}user: ${newMessage}`;
    }
}

// API 端點
app.post('/api/chat/:sessionId?', async (req, res) => {
    try {
        const sessionId = req.params.sessionId || uuid.v4();
        const { message } = req.body;

        // 取得或創建會話
        if (!sessions.has(sessionId)) {
            sessions.set(sessionId, new QSession());
        }

        const session = sessions.get(sessionId);
        const response = await session.sendMessage(message);

        res.json({
            success: true,
            sessionId: sessionId,
            response: response,
            messageCount: session.history.length
        });

    } catch (error) {
        res.status(500).json({ success: false, error: error.message });
    }
