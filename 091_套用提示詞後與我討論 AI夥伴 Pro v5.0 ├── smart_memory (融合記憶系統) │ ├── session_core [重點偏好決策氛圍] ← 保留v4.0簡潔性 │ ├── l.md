# 在本地用 Gemini CLI + Cursor（含MCP）完成「分類→處理→知識庫整合」的實戰方案

以下提供可落地的端到端流程：從資料匯出、清洗與分塊、主題/子主題分類、異質對話拆分，到本地建置向量索引與檢索問答（RAG），並結合 Google Gemini CLI 免費額度與 Cursor Agent（含MCP）來自動化批次處理與治理節點。每一步都附上可用的開源與配置要點。

## 目標與約束

- 目標：將上百篇 AI 對話與逐字稿、提示詞、技術驗證、家庭議題等，統一「分類→處理→知識庫化」，並可在本地檢索與問答。
- 約束：優先使用 Google Gemini CLI 免費額度；以 Cursor（Agent+MCP）協作；資料來源包含 Google Drive、Notion，最後轉移到本地存放與執行。[1][2][3][4]

## 整體架構總覽

- 採用兩層管線
  - 收斂管線（ingestion）：匯出→正規化→分段→主題/子主題分類→多主題拆分→品質標註與元資料。
  - 提供管線（serve）：本地向量庫（embeddings）→檢索（RAG）→問答→持續更新。
- 自動化中樞
  - Gemini CLI：批次摘要、分類、拆分、清洗、格式化（以免費額度大量處理）[2][3][5][6][1]
  - Cursor Agent：CLI非互動模式跑任務、管理多執行緒、讀取規則、串 MCP 來源（Drive/Notion/DB等）[4][7]
  - MCP：把雲端資料源（如Supabase或其他）或工具能力以標準協定掛進 Agent 流程，供讀/寫或查詢[7][4]

## 分步實施方案

### 第0步｜環境與帳戶配置

- 安裝 Gemini CLI
  - npm 全域安裝：sudo npm install -g @google/gemini-cli，或直接 npx 執行。[5]
  - 登入並啟用免費 Code Assist license（含2.5 Pro與100萬token context、60RPM、1,000 RPD預覽額度）。[3][1]
- 安裝 Cursor 並啟用 Agent CLI
  - 使用 cursor-agent，在專案根目錄加入 .cursor/rules、AGENT.md，並熟悉 --print、--output-format 與 --resume 等非互動/續航參數。[4]
- 設定 MCP
  - 在 Cursor/Agent 中配置 mcp.json；可連結如 Supabase MCP server 或其他服務，擴充工具存取資料源。[7][4]

### 第1步｜資料匯出與本地落地

- 來源
  - Google Drive：先以官方API或匯出腳本下載到本地（偏好純文字：.md/.txt/.jsonl）；若欲直接在流程內讀取，可在 Agent 內使用 MCP 對應的 server/工具（或以腳本前置拉取），但最終仍落地本地檔案以便統一治理。[7]
  - Notion：使用 Notion API 或現成導出工具，轉為 Markdown/JSON，保留頁面層級結構與屬性（tags、created_time、last_edited_time）。
- 統一命名與目錄
  - 建議結構：data/raw/{source}/{yyyy-mm}/{slug}.md、data/processed/{batch_id}/...、data/index/
  - 以來源與日期做可追蹤命名，為後續壓縮與回溯準備。

### 第2步｜正規化與清洗（Gemini CLI 自動化）

- 目標：去除雜訊、統一格式、保留必要元資料。
- 方法：
  - 使用 Gemini CLI 非互動模式與 --output-format json，批次處理每篇：
    - 任務1：規範化輸出（欄位：title、date、source、speaker_roles、content、tags_guess、language、pii_flags）
    - 任務2：刪除冗餘轉寫標記（例如逐字稿口癖）、保留技術代碼區塊完整
  - 實務做法：
    - 利用 cursor-agent 執行腳本，串 gemini 命令，將結果導出至 data/processed/{batch_id}/normalized.jsonl。[6][1][2][5][4]

### 第3步｜主題/子主題分類與多主題拆分

- 分類策略：
  - 先粗分主題桶（例如：AI技術/提示詞/逐字稿整理/技術驗證/家庭議題/其他），再細分子主題（如模型比較、RAG、MCP、工具鏈、Prompt工程模式等）。
  - 對同一文檔內的多主題段落，進行「語義段落切分」再各自分類，避免被單一主題標籤覆蓋。
- 工具與流程：
  - 用 Gemini CLI 完成初步分類與子主題抽取，並同時輸出每段的主題信心分數與不相干段落標示（flag irrelevant/misc）。[1][2][5][6]
  - 對於信心<0.8的段落，Cursor Agent 透過規則啟動二次判讀（或交叉比對關鍵詞/規則庫），並將難例回寫為人工複核清單。[4]
  - 產出：data/processed/{batch_id}/classified.jsonl（每段包含 doc_id、chunk_id、topic、subtopic、confidence、text、source_path）。

### 第4步｜分塊與元資料治理

- 分塊原則：
  - 技術段落：保留程式碼塊為一塊；長文本用語義斷句再以長度限制切段（例如800–1,200字/chunk），再輸出 overlap=100–200字的前後文橋接。
  - 非技術段落（家庭議題等）：優先語義分段，避免碎片化。
- 元資料欄位建議：
  - topic、subtopic、tags、source、created_at、edited_at、speakers、confidence、privacy_level（公開/內部/敏感）、pii_flags。
- 工具：
  - Gemini CLI 進行語義切分與標註；必要時用 Cursor Agent 對低信心段落呼叫二次策略（規則或小模型快速refine）。[2][1][4]

### 第5步｜本地向量索引與檢索（RAG）

- 選型：
  - 若需要開箱 UI 與本地模型整合：Kotaemon（RAG UI，支援Ollama、llama-cpp、可自訂pipeline）。[8]
  - 自建Python管線：faiss/chromadb + sentence-transformers/本地embeddings（Ollama nomic-embed-text 等）。[8]
- 流程：
  - 為 data/processed 的 chunk 生成 embeddings（可先用本地embedding模型；如需更高品質再考慮雲端）。[8]
  - 建立索引與檢索器；將 topic/subtopic 作為前置檢索條件或重排特徵。
  - 問答層：可接本地 LLM（Ollama的Llama/Qwen）或在限額內以 Gemini CLI 做生成層（將檢索到的上下文餵入，並要求引用來源片段）。[5][1][2][8]
- 補充：
  - 若想快速有UI試用：Kotaemon 指引支援Ollama與gguf/llama-cpp，能以本地模型運行與自訂管線，方便驗證RAG效果。[8]

### 第6步｜自動化、可觀測性與治理

- 使用 Cursor Agent 的非互動模式與規則
  - 用 --print 與 --output-format json 讓輸出可供下一步腳本處理；以 --resume 管理長對話上下文；以 .cursor/rules/ 與 AGENT.md 給代理上下文規範（例如分類標準、標註準則）。[4]
- MCP 整合資料源/工具
  - 透過 mcp.json 掛載可讀取來源（如Supabase MCP server，或其他MCP server）到 Agent 中，以標準協定調用；同時可設定只讀保護，避免誤寫。[7]
- 監控
  - 設計「低信心回落隊列」：confidence<0.8 的片段另存 pending_review.jsonl，定期人工或二次模型檢查。
  - 日誌：每批任務產生 manifest（處理數、錯誤、平均信心、主題分布）。

### 第7步｜安全與隱私

- 在清洗階段以 Gemini CLI 檢測 PII 與敏感內容，標記 privacy_level；RAG 層檢索時根據查詢者/用途過濾私密段落。[1][2]
- MCP 連線使用最小權限與只讀模式（Supabase指引也推薦預設只讀）。[7]

## 可用開源與工具清單（對應任務）

- Gemini CLI（免費額度、1M context、批次處理、非互動輸出、可結合搜尋與MCP工具）[3][6][2][5][1]
- Cursor Agent CLI（支援 MCP、規則檔、非互動模式、續航載入）[4]
- MCP（Model Context Protocol）：標準化工具/資料源接入，Supabase提供清楚整合示例，可類推其他服務[7]
- 本地RAG方案：Kotaemon（RAG UI、Ollama、llama-cpp、本地化與可擴充管線）[8]
- 向量庫：FAISS/Chroma（Python 套件）
- 本地模型：Ollama（llama3、qwen、nomic-embed-text等）[8]

## 任務樣板與命令範例（方向性示例）

- Gemini CLI 非互動批次（概念示例）
  - 對單檔規範化輸出：gemini --input-file data/raw/xxx.md --prompt "將內容清洗為JSON，欄位: title,date,source,content,pii_flags..." --output-format json --print > data/processed/{batch_id}/normalized.jsonl（管線中迭代跑多檔）[6][2][5][1]
  - 主題切分與分類：gemini --input-file data/processed/{batch_id}/normalized.jsonl --prompt "語義切分、每段topic/subtopic與confidence..." --output-format json --print >> classified.jsonl[2][5][1]
- Cursor Agent 串整段流程
  - cursor-agent -p "執行 ingestion batch: 掃描 data/raw/*，逐檔呼叫 gemini 正規化→分類→輸出 JSONL；對低信心片段輸出 pending_review.jsonl" --output-format json（以 .cursor/rules 定義策略與格式）[4]
- MCP 設定
  - 在 mcp.json 指定需要的 MCP servers（例如Supabase MCP server），授權PAT與只讀範圍，讓 Agent 在需要時查詢現有結構化資料或寫入審核隊列[4][7]

## 進階：質量提升策略

- 雙階段分類：第一階段粗分，第二階段針對混雜段落（multi-topic）做子句級再判斷，提高純度。
- 主題本體（taxonomy）迭代：把常見子主題沉澱成 taxonomy 檔（json/yaml），在 .cursor/rules 引用，讓模型對齊標準。
- 交叉檢索：RAG 查詢時以 topic→subtopic→semantic rerank，提升命中品質。
- 引用要求：生成答案時，要求模型列出引用的 chunk_id 與 source_path，以利可追溯。

## 何時選擇現成RAG UI（如 Kotaemon）

- 需要快速看效果、調參問答、比較不同本地模型或embedding時，Kotaemon 提供現成UI與本地部署指引，能直接綁 Ollama 與 gguf/llama-cpp，降低整合成本。[8]

# 全流程手冊補遺與強化版指引（第一階段：文檔處理與分類）

以下是通盤檢視前述討論後，補上的遺漏重點與強化建議，確保第一階段「掃描→提示詞抽取→清洗→分段分類→版本識別→報告」真正可在大規模文件上穩定落地，並順利銜接第二階段的RAG。

## 1) 批次與重試機制

- 批次切分與節流
  - 大量檔案時，批次執行（例如每批50–200檔），避免一次性耗盡資源或遇到單點失敗卡整批。
  - 設置最大並行數（concurrency）與速率限制（rate limit），特別是外部API的任務。
- 穩健重試
  - 對於任何解析/分類/清洗失敗的檔案，建立 retry_queue.jsonl，包含錯誤訊息與重試次數。
  - 三段式回退策略：同提示重試→切換備用提示→切換「保守模式」產出最小可用結果（至少能進 normalized）。

## 2) 嚴格的輸出Schema校驗

- 雙層校驗
  - 結構化輸出一律走 JSON，且用「Schema校驗器」在本地驗證欄位完整性（缺欄位直接標記 invalid 並入 retry_queue）。
  - 對關鍵欄位做類型與值域限制（例如 confidence∈、privacy_level∈{public,internal,sensitive}）。
- 容錯與最小可用
  - 若段級產出失敗，保底輸出檔級 normalized 並標註 need_segmentation=true，避免整檔丟失。

## 3) 格式相容與特殊類型

- 支援多格式
  - md/txt/json/docx/pdf優先；若有圖片/截圖（如對話截圖），先記錄檔名與路徑，延後OCR處理。
- 代碼與表格保護
  - 對 fenced code blocks 與表格（Markdown table/CSV片段）使用保護標記，避免被不當切斷。
  - 分段時，代碼塊以整塊為單位；表格可視長度判斷保留為整段或轉CSV片段附屬描述。

## 4) 逐字稿與口語內容的專用策略

- 逐字稿特徵判斷
  - 時間戳、說話者標記、口語填充詞密度→標記 transcript_like=true，優先走「逐字稿清洗模板」。
- 清洗與摘要
  - 保留內容事實與關鍵結論；去冗贅口語與重複語段；標示 speakers_roles（若可得）。
  - 以段級輸出摘要（chunk_summary）供後續分類與索引（第二階段RAG會更有利）。

## 5) 多主題拆分與離題判斷

- 多主題策略
  - 先語義斷句，再以子主題分類；將跨主題片段拆為多個 chunk，避免「整段被單一主題貼標」。
- 離題/雜訊處理
  - 對 misc_flag=true 的片段，保留於 segmented 中，但在 catalog/檢索時預設不納入；必要時建立 misc_pool 以供後續人工複覈或合適歸檔。

## 6) 版本識別與家族（family）管理

- 輕量版本偵測
  - 基於檔名相似、title近似、時間接近→生成 family_id。
  - 同 family 內計算段級語義相似度，標記新增/修改/刪除候選，但第一輪僅輸出 duplicates_or_family.jsonl，不立即合併，避免誤合流。
- 第二輪合併（可延後）
  - 若要合併，遵從「先並置後合流」原則，保留演進痕跡與差異標記（change_type與source_versions）。

## 7) 提示詞抽取與治理的加固

- 更嚴格抽取規則
  - 抽取指令前後200–400字上下文，判斷用途與風險情境，避免把「示例輸入」誤認為「可重用提示」。
- 去重與準則
  - 語義相似（餘弦相似度）>0.9 視為重複，0.8–0.9 視為近似候選；建立 canonical 與相似清單。
- 風險與禁用條件
  - 每個提示詞需定義 risk_notes，例如「不適用含大量代碼」、「不適用雙語混寫」、「不適用敏感資訊」等；在實際套用時先檢核禁用條件，再選擇 fallback 提示。

## 8) 分流決策透明度

- 決策記錄
  - routing_report.json 記錄每檔分流原因、信心與特徵訊號（長度、逐字稿指標、多主題可能性、代碼密度）。
- 手動覆寫
  - 可添加 overrides.json（以檔名pattern或family_id為鍵）手動指定走哪條路與使用哪套提示，方便少量例外快速修正。

## 9) 隱私與敏感資訊管理

- PII與privacy_level
  - normalized 階段先標 pii_flags 與 privacy_level；若偵測敏感（emails、phone、address、身份证/護照號等），在 processed/safe_view/ 另存一份脫敏版本。
- 權限隔離
  - 在未來RAG階段使用 privacy_level 決定是否可以被檢索/生成引用；現階段則用它決定輸出報告的展示範圍。

## 10) 品質監控與迭代回圈

## 11) Taxonomy（主題本體）策略

## 12) 日誌化與可追溯

### 对话 8

**问题**: 因為我補充一下我裡面有很多已經整理過的提示是只是版本不一樣這個部分你有考量進去嗎

**回答**:
# 已整理但版本不一的提示詞：治理與合併策略

有，這一點已納入流程設計；以下把「已整理但多版本」的提示詞治理策略說清楚，確保不會誤覆蓋、能保留演進脈絡，並在實際處理時自動選用最佳版本。

## 1) 提示詞版本識別與家族管理

- 指紋與家族
  - 為每個提示詞計算語義指紋（dedup_key）與抽取關鍵欄位（purpose、input_scope、expected_output_schema、style_tone）。系統根據指紋相似度與命名/時間線建立 family_id，將相似或同源的多版本聚為同一「家族」。
- 版本標註
  - 同家族內，保留 version 與 timestamp；把「語義差異」和「Schema/欄位差異」分別標記，避免僅靠字面差異判斷。

## 2) 合併與保留原貌的原則

- 先並置、後合流
  - 首輪不做破壞性合併：對同家族的版本並置展示，標記差異（added/modified/removed 字段與文本）。
- Canonical（主幹）與分支
  - 自動評分（見下）選出暫定 canonical 版本作為主幹；對語義或用途明顯分叉的版本標記為 parallel_branch，保留為「特定情境優選」。
- 差異對齊
  - 對 expected_output_schema 做欄位級 diff，補全缺漏並標註欄位變更理由（若能從上下文推斷）。

## 3) 自動選版（哪個版本被套用）

- 選版評分構成
  - 結構穩定度：是否具備完整 expected_output_schema 且能被JSON Schema校驗通過。
  - 適配度：purpose/input_scope 與當前文檔特徵（逐字稿/長文/含代碼/語言）匹配度。
  - 風險控制：risk_notes 與當前文檔條件是否衝突（出現禁用條件即降權或禁用）。
  - 歷史表現：該提示詞過往帶來的平均confidence、錯誤率與重試率。
- 選版策略
  - 首選 canonical；若當前條件觸發 canonical 的 risk_notes，則自動切換到最合適的 parallel_branch 或 fallback（保守模板）。
  - 選版結果寫入 prompt_linkages.jsonl（含 rationale 與評分細項），可追溯。

## 4) 版本合併與去重的細則

- 語義相似度門檻
  - >0.9 視為重複：合併為同一 canonical，收斂文案差異。
  - 0.8–0.9 視為近似：保留為同家族不同版本，待使用數據累積後再決定收斂或分支化。
- 欄位級差異治理
  - expected_output_schema 若不完全一致，以 canonical 為基準，對缺失欄位標記 optional，避免因小差異導致整體不可用。
- 風格與語氣
  - style_tone 差異較大視為 parallel_branch（同用途但不同語氣），在 CASUAL/TECH_DEEP 等模式下可能各有所長。

## 5) 多語與情境變體

- 語言標記
  - 每個版本記錄 language 與是否支援多語輸出；當前文檔語言優先匹配相同語言或可多語的版本。
- 情境變體
  - 對含代碼/表格/逐字稿等情境維度，保留對應的變體提示（如「技術長文+代碼保護版」與「逐字稿清洗版」），避免單一提示硬套所有情境。

## 6) 安全與禁用策略

- risk_notes 必填
  - 每個提示詞需定義禁用條件（例：大量代碼、雙語混寫、含敏感個資、格式不規則PDF等）；一旦命中，代理改走 fallback。
- 風險白名單/黑名單
  - 可在 overrides.json 指定某些 family_id 的特定版本只允許在特定資料夾或來源使用，避免誤用。

## 7) 觀測與迭代

- 提示詞報表（prompts_report.json）
  - 按家族與版本統計使用率、平均confidence、錯誤/重試率、禁用命中率。
- 漸進收斂
  - 每處理數批後，自動建議「把哪幾個版本合流為canonical」「哪幾個保留為情境分支」「哪幾個淘汰」，並輸出變更建議檔 proposals_prompts_merge.json。

## 8) 實作要點落地到產物

- prompts_catalog.jsonl
  - 增列：family_id、canonical_flag、branch_type={canonical|parallel_branch}、version、timestamp、usage_stats。
- prompt_linkages.jsonl
  - 增列：family_id、selected_version、selection_score、rationale、blocked_by_risk={true|false}。
- rules/prompt_mapping.json
  - 以 family_id 為主對應任務類型；版本選擇交由自動評分執行，除非 overrides.json 明確指定。

## 9) 與文檔處理流程的整合位置

- 抽取與治理時機
  - 在「全量掃描」之後、分流之前先抽取與家族化，確保分流決策可用上「提示詞可用性」訊號。
- 在清洗/分段/分類中的應用
  - 實際套用時，先做禁用條件檢查→跑選版評分→選中版本→執行並寫入 linkages 與結果品質評分。

**回答**:
# 白話版流程說明與會達到的效果

## 整體想法
把上百篇不同來源、不同版本、混雜主題的內容，先「全部自動看過一遍」，再「自動決定怎麼處理」，最後輸出成一套乾淨、可用、可追溯的資料集，為之後做RAG或其他進一步應用打好地基。

## 流程一步步是怎麼做的

- 把資料全部丟進一個資料夾
  - 不用先挑或整理，md/txt/docx/pdf都可以，先「全吃」。

- 自動掃描與建檔
  - 系統會幫每個檔案做「體檢」，例如：長度、有沒有程式碼、像不像逐字稿、可能是同一系列的不同版本等，並記錄結果。

- 自動找到「提示詞」並整理成庫
  - 文檔裡若藏著「整理文章用的提示詞」或「逐字稿處理指令」，會被自動抓出來。
  - 相似或同家族（不同版本）會整合：不會亂覆蓋，而是保留為同家族，選出一個主幹版本（canonical），並把其他版本當作情境備選（例如技術文、逐字稿、含程式碼的情境）。

- 自動決定用誰來處理
  - 短／單一主題／格式整齊：用本地規則快速清洗與格式化。
  - 長文／逐字稿／混合主題：用較強的清洗與「語義切分+主題/子主題分類」，輸出更結構化的結果。
  - 決策過程會被記錄，方便之後檢視為什麼這樣分派。

- 清洗與標準化（normalized）
  - 內容去雜訊、保留程式碼原樣、統一標題與日期、加上基本欄位（來源、語言、是否敏感、品質備註等）。
  - 每一篇會變成一筆可讀的標準JSON記錄，方便後續任何工具使用。

- 語義切分與分類（segmented）
  - 文章會依內容意思分段；含程式碼的段落不會被錯切。
  - 每段會貼上主題/子主題與信心分數，離題的內容會被標註為「misc」。
  - 信心不夠的段落會列到「待複核清單」，避免誤分類。

- 版本識別與「家族」管理
  - 相似標題、相近時間、內容接近的檔案會被歸成一個家族（family），方便之後做版本合併或比對差異。
  - 第一輪不強制合併，只做識別與報告，避免誤傷。

- 提示詞與文檔的對齊與自動選版
  - 系統會把「哪個提示詞適合哪類文件／哪種段落」自動配好，並根據語言、是否逐字稿、是否含程式碼等條件自動挑選該家族裡最合適的版本。
  - 如果命中提示詞的「禁用條件」（例如含大量代碼不適用），會自動換用備用版本或保守模板。

- 產出報表與目錄
  - 總覽主題分布、低信心比例、哪些檔要重試、哪些提示詞最常用且效果最好。
  - 所有結果都有來源與位置可追溯，之後引用或檢查都方便。

## 實際上會得到什麼效果

- 效率大幅提升
  - 不用人工挑樣本、不用手動分類，系統能自動「看過、判斷、清洗、切分、分類」，把混雜的資料變成乾淨的結構化輸出。

- 資料品質變穩定
  - 有統一欄位與格式、每段都有主題/子主題與信心分數；低信心會自動列出來，降低錯誤流入後續流程的風險。

- 提示詞不會混亂
  - 版本多也不怕：會自動歸類成家族、保留演進歷史、在每次處理時自動挑最適合的版本，避免亂套或覆蓋。

- 可追溯、可擴充、可治理
  - 每個段落都能追到來源與原文位置；任何分類或清洗決策都有紀錄；需要人工覆寫也能加規則快速修正。

- 為RAG打好地基
  - 分段內容、主題標籤、摘要與敏感資訊標記都準備好，後續要接任何RAG框架或向量庫，都能直接拿 segmented 的結果去建索引。

- 隱私與安全更安心
  - 在清洗階段就會標記敏感資訊與權限等級，之後做檢索或引用時可以套用遮罩與過濾。

## 簡單講，整套流程的價值在於：
- 把「雜亂的大量資料」自動變成「乾淨、可用、可維護」的知識素材。
- 把「提示詞版本地獄」變成「有家族、有主幹、有情境備選」的可控系統。
- 把「之後RAG會用到的關鍵欄位」一次備齊，讓未來切任何技術路線都不被綁住。

如果要開始，最小步驟就是：
- 把所有原始檔案丟到 data/raw/
- 放上流程規則與任務說明
- 一鍵啟動自動處理

**问题**: 是沒錯但是好像漏掉了有沒有辦法自動連接剛剛講的那兩個平台上的文章幫我把那些文件弄下來

# 自動連接並抓取 Google Drive 與 Notion 內容的做法（可併入現有流程）

下面給出「零手動挑檔」的一次到位做法：用 Cursor Auto Agent 驅動，透過官方 API 把 Google Drive 與 Notion 的文章自動同步到本地 data/raw/，再銜接前面已定義的清洗、分段、分類與提示詞治理流程。整體保持可追溯、可重跑、可增量。

## 整體策略
- 採「本地為準」的鏡像同步：先把雲端文件拉回本地，落地成可處理的純文字或Markdown，再進管線，確保可離線治理與重現。
- 使用官方 API 與穩定格式：
  - Google Drive：Drive API + Docs API（將 Google 文件轉為 Markdown/HTML/純文字），支援增量同步與更改追蹤。
  - Notion：Notion API 將 pages 資料轉為 Markdown/JSON，保留屬性與階層。

## 一、Google Drive 自動抓取

- 權限與設定
  - 建立 Google Cloud 專案 → 啟用 Drive API（必要時加 Docs API） → 建立 OAuth 或 Service Account 憑證。
  - 憑證以環境變數或本地 credentials.json/ token.json 方式保存，避免硬編碼。
- 同步策略
  - 過濾條件：mimeType、擁有者/共用資料夾ID、標籤或檔名規則（例如只抓 AI 相關資料夾）。
  - 格式轉換：
    - Google Docs → 轉 Markdown（優先）或 HTML，再存為 .md；同時導出原始 JSON 結構以備對照。
    - 其它可直接下載的檔（.md/.txt/.pdf）：原樣保存。
  - 增量更新：
    - 使用「修改時間」與「Files: list changes」或比對 ETag/MD5，僅下載或更新變動檔。
- 落地結構

## 四、可直接採用的抓取腳本設計重點

- Google Drive 抓取腳本（Python/node 皆可）
  - 參數：folder_ids、since_timestamp、export_format=md/html/txt、include_mimeTypes

## 五、和既有管線的銜接點

  3) 分流（本地清洗 vs 強化清洗與分段分類）

  5) 版本家族識別與報告

## 六、權限與範圍控制

- Google Drive

- Notion
  - Integration 只共享必要的 Databases/Pages；可用「鏡像資料庫」方式隔離非必要內容。
- 最小權限原則 + 專案內 secrets 管理，避免憑證外洩。

## 七、錯誤處理與重試

- 無法轉檔或權限不足 → 記錄於 fetch_errors.jsonl，並輸出建議處置（需要授權/改格式/手動下載）。
- 下載中斷或部分檔案失敗 → 進 retry_queue.jsonl，限定最大重試次數與退避策略。
- 任何抓取與轉換失敗，都不阻塞後續其他檔案處理。

## 八、執行順序建議（白話）

- 第一次跑：全量抓取（Drive/Notion）→ 本地建庫 → 跑掃描與處理 → 出報告。

- 需要時可針對某個資料夾或資料庫做「立即重跑」。

標題：掃描、分流、清洗、分段分類與報告的規則

- 規則：
  - 短/單主題/格式整齊→本地清洗（簡路）
  - 長文/逐字稿/混雜主題→強化清洗+語義切分+主題/子主題（強路）
